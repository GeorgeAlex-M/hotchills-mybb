    Chapter 4

    Test Techniques

    4.1 Categories of Test Techniques

    Definition - Test technique is a procedure used to define test conditions, design test cases and specify test data.

    That means test technique provides you way using which you can define your test case and you can design your test case and you can specify test data to support your test case. 

    The choice of which test techniques to use depends on a number of factors:

    1. Component or system complexity

    Depending on how complex your system is the type of testimony you will use for varies

    2. Regulatory standards & Customer or contractual requirements

    Depending on what we promised to the customer and what all standards are applicable to our product. We have to use test technique to satisfy it

    3. Risk levels and types

    Depending on what is the risk associated with your product and the type of risk, you select the test technique. For example for banking system we need a highly secure system to avoid hacking of the user account.

    4. Available documentation & Tester knowledge and skills

    Sest technique also depends on which all supporting documents you have for your project and then how you use that information depends on your knowledge and skill.

    5. Available tools

    Different organizations use different tools for testing and each tool has its own specific features. So you have to use your knowledge and skill to see which test technique you can use so that the available tool can be used efficiently.

    6. Time and budget

    We have to select the test technique keeping in mind that using selected test technique we can complete the project in time and within the available budget. 

    7. Software development lifecycle model

    We also have to see that selected test technique is in line with the software development model we use.

    8. The types of defects expected in the component or system

    Different test techniques can be more effective to different types of defects.

    4.1.1 Categories of Test Techniques and Their Characteristics

    Test techniques are categorized into three categories:

    - black box test which is also known as behavioral or behavior based technique
    - white box testing which is also known as structural or structure based technique 
    - experience based testing where testing depends on the experience of the developer and tester

    Black-box

    Definition - Black box testing is based on an analysis of the appropriate test basis and it concentrates on the inputs and outputs of the test object without reference to its internal structure.

    So in black box testing we analyze the requirement and then we see what are the output and what are the inputs. We don't care about the internal structure.

    Example:
    Code:
    Requirement:
    If the speed of the motor is more than 150km/h and temperature value is greater than 120 degrees then switch off the motor


    We saw this example during system testing explanation. Here we had this requirement. In black box testing we concentrate on what are the inputs. In this example the inputs are speed more 150km/h and temperature more than 120 degrees. Along with input we also see what is the output. Here the output is that motor will stop so we have two inputs and one output. This is what we need for black box testing. Why it is called black box? Because for the tester, the internal structure of the code is not visible only input and output is visible. 

    White-box

    White box testing which is opposite of black box testing. 

    Definition - White-box testing is based on an analysis of the architecture, detailed design, internal structure or the code of the test object. And it concentrates on the structure and processing within the test object.

    Here you can see we care more about the internal structure of the code. 

    Example:
    Code:
    Requirement:
    If the speed of the motor is more than 150km/h and temperature value is greater than 120 deg then switch off the motor




    Let's understand this by using same example. Supposed to achieve this functionality. If this is the piece of code written, then for white box testing divide them into small units and then we see that the code is implemented correctly by performing white box test techniques on them. This technique is named white-box because we can see the code here.

    Experience-based

    Definition - Experience based techniques are often combined with black box and white box test techniques.

    Here, the tester performs testing based on his experience and skill and not based on test basis. 



    Now let's see some common points and differences among these test techniques: 

    - Test conditions, test cases and test data are derived from a test basis that may include.

    - For black box testing we can write test cases from software requirements, specifications, use cases and user stories

    - For white box testing we need code, software architecture, detailed design or any other source of information regarding the structure of the software

    - For experience based testing we don't need requirement or code, here we write test case based on knowledge and experience of testers, developers, users and other stakeholders

    Next point is related to coverage

    - For black box and white box testing we can provide coverage but not for experience based, because we don't use any document for performing experience based testing

    Description of test techniques are provided in ISO/IEC/IEEE 29119-4

    4.2 Black-box Test Techniques



    There are five different types of black box testing equivalence partitioning, boundary value analysis, decision table testing, state transition testing, test cases from a use case.

    4.2.1 Equivalence Partitioning


    Definition - Equivalence partitioning divides data into partitions (also known as equivalence classes) in such a way that all the members of a given partition are expected to be processed in the same way.



    Suppose you have this requirement. That means the system will accept values from 10 to 99. But is it enough to test the values? The answer is no. Here we have to apply equivalence partitioning technique. We can represent data like this. Where number 10 and 99 are kept at the boundary as these are the extreme values, which system accepts. Now, if you see, we have three partitions values less than 10 which are invalid, values more than 99, which are also invalid, that means here we have three partitions. We can also say that we have three equivalence classes. Two invalid equivalence class and one valid equivalence partition.

    Summary:

    - Valid values are values that should be accepted by the component or system. An equivalence partition containing valid values is called a valid equivalence partition.

     - Invalid values are values that should be rejected by the component or system. An equivalence partition containing invalid values is called an invalid equivalence partition. 

    - Any partition may be divided into sub-partitions if required. What it means is in the valid equivalence partition we can have multiple partitions. 

    - Each value must belong to one and only one equivalence partition.

    - One invalid equivalence partitions are used in test cases. They should be tested individually.

    NOTE: Testing all the values means that you are exhaustive testing, and we know that exhaustive testing is impossible.

    4.2.2 Boundary Value Analysis

    Definition - Boundary value analysis is used to test the values that exist on the boundaries of ordered equivalence partitions. 

    Suppose you gave a set of data then at the boundary, you have to apply the boundary value analysis. Before going further, you need to know that there are two approaches by which we can apply boundary value analysis. 

    - Two value boundary testing
    - Three value boundary testing

    If nothing is mentioned, then you have to apply two value boundary testing. 



    Suppose this is the requirement. An input field accepts a single integer value as an input, and the valid range is from 1 to 5. Here is clear that 1 and 5 are the boundary values. Now let's see how to apply two value boundary testing and three value boundary testing. For two value boundary testing, you need to take two values outside the range, here we have to take zero and six. For three value boundaries testing, you need to consider two values outside the range and two values inside the range. First, we need to take zero and six, and then we need to test two and four. Now, if you look at both the diagram, you can clearly see why they are named two value boundary testing and three value boundary testing. Here at each boundary we are testing two values. One at the boundary and one outside the boundary, and here at each boundary, we are testing three values, one at the boundary, one outside the boundary and one inside the boundary. 



    Suppose the requirements is field except the values 1 to 10 in increments of 0.5. For this requirement, this is our boundary analysis diagram. Now let's draw to value boundary testing diagram. Since the increment value is 0.5 we need to take one value outside 1 and it will be 0.5. Other value is 10.5. Now let's move to the three value boundary testing. Here outside value is zero point five. And inside value is one point five. And here the outside value is 10.5 and inside value is 9.5.

    4.2.3 Decision Table Testing

    - Decision tables are used to test the interaction between combinations of condition. 
    - The goal of Decision Table testing is to ensure that every combination of condition, relationship and constrains is tested.
    - When we're trying to test every possible combinations, decision tables can becomes very large.
    - A method of inteligently reducing the number of combinations from all possible to those which are "interesting" is called collapsed decision table testing.

    Collapsed decision table

    Normally the collapsed decision table is the table which is drawn after the analysis of the main table. 

    - When this technique is used, the combinations are reduced to those that will produce differing output by removing sets of conditions that are not relevant for the outcome.
    - Redundand tests or tests in which the combination of conditions is not possible are removed
    - Decision whether to use full decision tables or collapsed decision tables is usually risk-based

    The common notation in decision tables is as follows

    For conditions (input):

    - Y means the condition is true (may also be shown as T or 1)
    - N means the condition is false (may also be shown as F or 0)
    - - means the value of the condition doesn't matter (may also be shown as N/A)

    For actions (output):

    - X means the actions should occur (may also be shown as Y or T or 1)
    - Blank means the action should not occur  (may also be shown as - or N or F or 0)

    4.2.4 State Transition Testing

    Definition - State transition testing is used to test the ability of the software to enter into and exit from defined states via valid and invalid transactions

    So when we're performing a state transaction testing. Our objective is to see how can we enter into some stage or how can we come out of the stage and to do that we have to perform a valid as well as invalid transactions.

    - We are going to verify different states and when I say different state that means valid or invalid state and valid and invalid transactions. 
    - The goal is to state test the valid state tests as well as invalid states. 
    - We are performing these tests so that we can cover the sequence of states

    So whatever sequence is defining the requirement that sequence we can test by using state transition testing.

    Example:
    Suppose you want to log in. When you're logging in there are two states. First one is you can successfully log in and browse the data, the other one could be error message but to achieve these two different states you have to perform different actions or different events. If you perform a valid event that means your user I.D. is correct and your password is correct then you are able to browse the data whereas the other one could be invalid transaction where your login is incorrect and your password is incorrect or both are incorrect. In that case you will learn it from the error message. There could be one more state or one more event when you are browsing the data, timeout happens you are not doing anything you're idle in that case also you may come out of the browser and that will also result in an error message. So these are the different dates and data. That is how you transit between these dates. 

    So from this example we can have some analysis like:

    - Events cause the software to transition from one state to another and perform different actions. 

    So we were in the log in state and we have these to provide the correct user I.D. as well as password. If that event is valid then we enter into browsing data and if that event is wrong then we enter into error message. 

    - Events may be qualified by conditions (sometimes called guard conditions or transition guards) which influence the transition path to be taken. 

    Sometimes the event can be negative or positive that means valid or invalid and will result in different conditions so that we take different paths whereas we are browsing the data or get the error message to pop up.

    4.2.5 Use Case Testing

    1. Definition - Use cases are defined in terms of interactions between the actors and the system that accomplish some goal

    Suppose there is an ATM machine and there is a user. So he goes to the machine in order to draw some money out of it. If he's standing here he cannot draw the money. So he has to interact with the system and then only he can get some money out of it. And here the goal is to get the money out of the ATM machine. 

    2. Provides - Scenario-based tests that should emulate usage of the system

    3. Applicability - At the system and acceptance testing levels



    So first we have to understand who are the actors to your user and the system. Here user and system are the actors. Here the user has a card which is easytravel card. So he's putting that particular Travelcard on a machine. So when he's doing that activity that putting the card on the machine then the machine is asking him three questions. Whether he wants to check the balance, add the balance to the card or he wanted to see the transactions. Now if you observe here you will see something different. 

    There is a bracket and inside it is says separate use case. But for the B option there is nothing written and that means this use case is going to be for Option B that we going to add a balance here and if you want to protect other two options then there is a separate use case for it. But right now we will discuss about how to add a balance to his easytravel card. 

    So now user is selecting add balance. So system will ask how much balance you want to add then the system will select the amount then the system will ask how you want to pay. You want to pay cash or you want to pay by a credit card. Then user select the credit card. Then system will ask you to insert the credit card. Then what user will do he will insert the credit card. Then the system will show the amount of how much he is adding and then the system will ask him for the confirmation, after that the user will confirm it and the amount will be credited into the easytravel card and debited from the credit card. And then the user will remove the credit card as well as the easy travelcard then the system will print that transaction. And at the end system will return to the main menu.

    In this particular example or in this use case testing you have seen that how the interaction between the user and the system is happening. It's a very close interaction. Once the user is doing an action then the system is doing aswell. Now this is how use cases are written.

    4. Actor - Human users, external hardware or other components or systems

    Actors could be a human user or external interface or it could be external hardware or external system. 

    5. Includes - Possible variations of its basic behaviour, including exceptional behaviour and error handling

    When the system is asking to select the amount and if the user is not selecting and he's just leaving, then system has to come to the main menu or he has to do some other activity. What will happen if the system is asking the user to insert his credit card but the user is not inserting the credit card. So these are different type of scenarios for which we have to again write the use case testing. 

    6. Described - By interactions and activities, preconditions, post-conditions and natural language where appropriate


    So what we are seeing is one step user will do and one step the system will do. So these interactions and the activities then the preconditions post-conditions and some neutral language. So then you go in front of the system there will be some precondition that there would be a welcome window and once you complete that activity there will be a thank you window. So preconditions and post-conditions are also part of use case testing so it is described by interactions and activities.

    By performing test case testing

    - We are verifying the process flow of the system (how the system should behave, how the control should flow in the system)
    - The goal is to check likely use of the system in which our system will be used
    - The significance is to find defects due to the interaction of many components (user, system, etc. So all these components when coming together what type of defect we will get)

    4.4 Experience-based Test Techniques

    - Experience-based is a technique in which the test cases are derived from the tester's skill and intuition and their experience with similar applications and technologies. 

    That means in this type of testing the success of testing entirely depends on the testers skill and knowledge.
    We saw how test cases derived from the test basis. First we analyzed the test basis. Then we derived test condition. And finally we write test cases. But in experience based testing we don't derive test cases from test basis. Here the tester writes new test cases based on his experience and skill. 

    - These techniques can be helpful in identifying tests that were not easily identified by other more systematic techniques. 

    Since the tester knows how the system works and based on his experience he may also know what types of defects were detected before so this will help in writing effective test cases.

    - Not suitable to provide requirement coverage

    Since the testing is not based on requirement. It is hard to provide coverage in experience based test techniques. 



    Experience-based technique is of three types. Error guessing, exploratory testing and checklist based testing. 

    1. Error guessing

    Here, tester based on his experience and knowledge thinks about possible errors, defects and failures. He thinks how the application has worked in the past. What kind of errors tend to be made, failures that have occurred in other applications. And then list them down to create the test design or test case. An error guessing technique which is a type of experience based technique, the test case is not written based on test basis. The quality of test case completely depends on the knowledge of the tester.

    - Error guessing is a technique used to anticipate the occurrence of errors defects and failures based on the testers knowledge. 
    - While thinking, we ask these questions, how the application has worked in the past? What kind of errors tend to be made? Failures that have occurred in other applications. 
    - Error defect failure lists can be built based on experience, defect and failure data or from common knowledge about why software fails.

    2. Experience based

    Is exploratory testing and it also is a dynamic testing. In this testing the tester while test thinks about the test case based on how the product works to do this. He uses his experience and knowledge. Here the test cases are not formal. Sometimes they are documented and sometimes not. And then immediately these test cases are executed. Here tester thinks about different combinations and dynamically test team. 

    - In exploratory testing, informal not predefined tests are designed, executed, logged and evaluated dynamically during test execution 
    - Exploratory testing is most useful when there are few or inadequate specifications or significant time pressure on testing
    - Exploratory testing is sometimes conducted using session-based testing to structure the activity

    3. Checklist-based testing

    Where the checklist contains test conditions which need to be tested. Here, the tester test the feature based on the checklist provided while testing based on testers knowledge. If he feels more points can be added to the checklist then he adds more points to it and tests. This is how experience based testing is combined with checklist based testing. 

    - In checklist based testing, testers design, implement and execute tests to cover test conditions found in a checklist.
    - As part of analysis, testers create a new checklist or expand an existing checklist., but testers may also use an existing checklist without modification 
    - In the absence of detailed test cases, checklist-based testing can provide guidelines and a degree of consistency.
