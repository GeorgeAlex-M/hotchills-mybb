    A software is a written code that applies to a specific environment in order to automate the process.

    There are two types of software systems:

    Business Applications
    Customer Products

    Examples of Business Applications:

    When you go to the airport to book your ticket,
    When you go to the supermarket to buy items
    When you are working in the office using an application
    When using a website

    Examples of Customer Products:

    Washing machines
    Mobile phones
    Coffee makers
    Cars



    Software testing is a way to assess the quality of the software and to reduce the risk of software failure in operation.

    Once the developer develop the software, before giving it to the user, the software undergoes a process to find out the defects and risks associated with the software. Therefore by finding defects we increase the software quality and reduce the risk associated with the software (such called failure in operation).



    Now we all might gues that a tester is a physical person who is involved into a quality control activity. What most of the people before enrolling into a tester job is that they do not recognize the drawbacks or they just omit to think about them. Before getting to the heart of it for now we need to know that a tester is involved into a quality control activity and the most important aspect for a tester is the time associated with the test activity, and most of the time a good tester will know how to manage his test activity using particular methodologies to grab as much time as he can and this goes on in a cycle, the company will be happy having their projects to be finished faster while the costumer will also be happy having their software delivered in a shorter time than it’s deadline which will also improve costumer relationship and confidence and basically having lots of projects means more money for the company you are working for.

    Automation on the other hand is a script written (usually in Python) by the tester which will be executed on the areas that produce most of the bugs, and the same concept like above applies here as well, writing efficient test cases will give us more time to use somewhere else.

    My recomandation to you is that if you are a junior tester you should focus on using particular methodologies that you've learned when you design your test cases, because just being good at finding issues in the code comes with experience in the project you are working. Those methodologies are designed to cover and grab as many possible issues, so using tested and approved methods will get you on the right path.

    Now as you have an idea of what is going on there is much more to be discussed about testers and automation which will be covered in the upcoming chapters.



    A software should work as expected whenever we use it. To ensure this, testing of the software is required.

    A software that does not work correctly can cause a failure in its operation which results in loss of the company's money, time, reputation and in extreme cases even injury or death of the end user.

    The probe concluded that the 2012-2018 Model S and the 2016-2018 Model X vehicles “contain a defect related to motor vehicle safety” reported Reuters.

    Misperception I
    Testing only consists of running tests, i.e., executing the software and checking the results.

    Answer:
    Testing is a process and consists of many activities : test planning, test monitoring and control, test analysis and design, test implementation and execution, test completion. (we will discuss about each of them in the upcoming chapters but for the time being you need to remember these activities)

    Misperception II
    Testing only involves the execution of the component or system being tested.

    Answer:
    Testing consists of both dynamic and static testing. In dynamic testing a code is executed whereas in static testing a code is not executed.

    Explanation:
    Dynamic testing is when you are physically executing the code. For example the developer has written a block of code and he wants you to execute it to see if it works as expected.

    When performing static testing you look onto the developers code and check out for errors. For example the bulb shall glow only when the door is closed:
    Code:
    if door.state ~= true then
        bulb.light = true
    end

    Lets understand this piece of code, the condition is if the variable for checking if the door is closed is door.state while ~= is called operator and compares if the door state is not true while variable bulb.light sets its value to true (true or false, 1 or 0 are called booleans). The translation for this lua code block is if door is not closed then light the bulb. From this we can notice that the developer slipped the operator ~= instead of == and the right code should look like the above

    Code:
    if door.state ~= true then
        bulb.light = true
    end


    Misperception III
    Testing focuses entirely on verifications of requirement.

    Answer:
    Testing focuses on both verification and validation of the requirements.

    Verification means are we building product right?

    Validation means to check if we built the right product.

    We will discuss of both validation and verification and explain the difference between them in the upcoming chapters.

    1.2 Why is testing necessary?

    In order to understand this we must remember that all of us are human and being human we make mistakes and those mistakes can be very expensive, the expense can be loss of money, it can be a crucial loss of time, it can also be a loss of business reputation and the final and gravest loss is death or injury due to our mistake. In order to avoid these loses and minimize risk we have to test every single aspect of our product.

    Points for why is testing necessary:
    • Reducing the risk - We perform testing to reduce the risk associated with the product.
    • Detecting the defects - We have to detect the defects so they are not seen in operational use. So when the user is using our product they shall not be able to find these defects. Its our responsibility to find them first through testing.
    • Meeting the contract - We have to make sure that we are meeting all the commitments made to our customers. So if the customer asks for something that’s not included in the contract this is where we find it.
    These are the reasons testing is necessary to the process of product development.

    Example:
    There is a developer who works on a code and creates a software. He then directly hands over that software to the customer. Now that the customer has the software he uses it but soon becomes very disappointed because he has found an error in the software. So now the question arises why did the customer find the error and not us. The answer is that once the software was developed we handed it over directly to the customer instead of going through this rigorous process of testing. If we had put the product in this process it is possible that we could have caught the error before it ever reached the customer and this is where I have to mention something crucial. We have to perform appropriate testing at appropriate levels. The development of the product can be broken down into several levels of activities and at each level we have to decide what is the appropriate testing that should be carried out.

    Levels of development activities:

    Requirements - How testing requirements contribute to success.
    Design stage - How testing design contributes to success.
    Coding - How testing coding contributes to success.
    Software - How testing software contributes to success.

    If we carry out tests at each level then we will achieve a successful product.

    1.2.1 Testing’s contribution to success

    Levels of testing:

    Requirement reviews I:

    In this level we start by reviewing the requirements. The reviewing is nothing but static testing. So we are performing static testing on the requirements. We are doing this to detect the defects in these work products. Here of course the work products are just the requirements. We are reviewing these requirements to reduce the risk of incorrect or unstable functionality being developed. Which means if there is some wrong or incorrect in the requirement and we don’t spot it then that is how the entire project will be developed. This is what we have to stop by performing testing on requirement.

    Example:
    Let’s have a look unto the software development levels in ascending order:

    1. User requirement
    2. System requirement
    3. Global design
    4. Detailed design
    5. Implementation
    If we do not correct the error at the system requirement level it becomes part of the global design and the detailed design, both of which will be wrong and eventually we will have a software that doesn’t work correctly.

    So that is why if we correct the requirement at this stage the rest of the process will carry out without any errors due to requirement. This is why testing the requirements contributes to the success of product.


    Design II:

    At design level there will be a system design and our job is to do a review on it. Since there is no code so we can’t run it. Therefore we review it which is a static testing technique where we do this review it will increase team members understanding. So whether is the tester, developer or the process guys everyone will have the same understanding after review. If everyone has the same knowledge it will help reduce the risk of fundamental design defects. So a review of the system design will reduce the risk of fundamental design defects.

    Example:

    1. User requirement
    2. System requirement
    3. Global design
    4. Detailed design
    5. Implementation

    We are now we are at the global and detailed design stage. If we carry out the testing here and catch the mistake then it will not go on to the implementation stage. We will send the correct software for the dynamic testing and product will be a success. If we don’t do the testing then we will send a faulty software for dynamic testing and then we will be forced to go back to stepts to find the error.

    Conclusion is if we perform testing at the design level it contributes to the success of the whole project.

    Coding III:

    In the coding level you have code under development. The code is being developed and you will perform the static testing on it. The intention with this is to increase the understanding of that piece of code. What this does is reduce the risk of defects within the code. So if there are any defects we can find them out by reviewing the code through static testing.

    Example:

    Customer requirement - Check that the window will move up with in 10ms once the button is pressed.


    Code:
    UpMovementFunc()
    {
    if(PressTimer <= 10)
    {
    WindowMovement = Up;
    }
    }

    The developer has written if the button is pressed for 10 milliseconds or less then the window will move up. The requirement was for the window to move up within 10 milliseconds not at 10 milliseconds. The developer has included the less < operator which changes the result, now the window might not move up unless exactly 10 milliseconds have passed.

    This is how performing testing at the coding level can reveal mistakes in the code and contribute to the success of the whole project.

    Software IV:

    Once the software is ready we verify and validate the software. Now that the software is functional and the codes are ready we can perform dynamic testing on it. Once we perform dynamic testing we detect failures that might otherwise have been missed. If we don’t do the testing at this level then there is a fair chance that we will miss some errors and they will finally be discovered when the software is in operational use. Next is removing the defects that cause the failures so we will fix the problems because of which the software is failing. This is where the debugging process comes into play. The tester finds the error and tells the developer who fixes the problem and sends it back to the tester to confirm that the error is gone. And by doing all of this our software will meet the stakeholders needs, it will meet the customers or the users requirements and that means everyone using the software will be happy.

    The first point of difference is static testing is done without execution of code.
    Whereas in dynamic testing, we are executing the code.

    Static Testing

    Example:

    Let us say that we have a customer requirement that states window shall move up when the up button is pressed and shall move down when the down button is pressed. To fulfill this requirement, you have to write a code for it. Now, suppose a developer has written this particular code. 

    Code:
    UpMovementFunc()
      {
        if(PressTimer <= 10)
        {
          WindowMovement = Up;
        }
    }

    This is a function where he's trying to implement the customer's requirement. When Button is pressed if button up, then move up. Otherwise, move down. Our task as a tester is to check whether this piece of code is satisfying the requirement or not. And this type of testing is known as static testing. So let's try to test this particular code here. It's saying that if a button is pressed, then move up, but if up button is not pressed, then move down. If you execute this code, then the window will never stop moving. It will move up. If you press the up button. But the moment you release the button, it will keep on moving down. This means that this code is failing to fulfill the requirement. So this is how we find defects through static testing. So why do we call this static? Because when I was performing this test, this code was not moving. It was in one place. This is why this type of testing is called static testing. So that's it for static testing.

    Dynamic Testing

    Dynamic testing requires you to study the behavior here. If you press the down button, then the window moves down. You aren't bothered about the code or how it is written. When you press the up button window will move up. And when you press the down button, it will slide down. Here, what you need to remember is static testing is carried out without the execution of code, while dynamic testing requires execution of code.

    Difference between Dynamic and Static Testing

    Difference I

    Static testing is conducted in the verification stage, whereas dynamic testing is performed in the validation stage.

    Let's find out what they mean by verification stage and validation stage.

    So here we have the lifecycle of a software development activity:

    First, you will have a user requirement.
    Next, a system requirement.
    Then a global requirement.
    Then you will have a detailed design.

    1. User requirement
    2. System requirement
    3. Global design
    4. Detailed design
    5. Implementation
    Once you have this design, then you can start writing the code. Now, static testing is done in the early stage when we only have the document.

    We don't have the code in the User Requirement, System requirement, Global design and Detailed design stages. And if we don't have a code, we can't run a software. When we have only documents, then we can only perform static testing. We can carry out a verification but we can't run the code.

    So these are the two stages. The first part is for static testing (User Requirement, System requirement, Global design and Detailed design stages). But once the code is ready, you can perform the dynamic testing (Implementation stage) because now you can run the code. This is why static testing is done in the verification stage, whereas dynamic testing is done in the validation stage.

    Difference II

    So once you have the code available, you can perform the dynamic testing. Otherwise, it's not possible.

    Difference III

    Static testing is cost effective, whereas dynamic testing is less cost effective.

    Example:
    1. User requirement
    2. System requirement
    3. Global design
    4. Detailed design
    5. Implementation

    As we saw in this diagram, let's assume you were in the requirements stage and you found a defect here. Since you found the defect in the same stage, cost to fix the defect is less. But during dynamic testing, you are all the way over at the Implementation stage. And if you find a fault here and after analysis, you've found that the requirement is wrong. Then you have to change all these documents (user requirement, system requirement, etc... all the way to the implementation document). This will require more people to solve problems and costs to fix defect will increase. So this is the reason we said that static testing is cost effective while dynamic testing is less cost effective.



    The first point is static testing is conducted without execution of code, dynamic testing requires execution of code.

    The second point is static testing is performed in the verification stage, whereas dynamic testing is performed in the validation stage.

    The third point is static testing is cost effective, while dynamic testing is less cost effective.

    There is a fourth point about examples that I want to add here:
    Static testing examples are Walk Through and code reviews, whereas with dynamic testing, you have to perform functional or nonfunctional tests.

    Before we jump to the objectives, you must know these important terms which we are going to use and these are work product and test item.

    Work product means output.

    Let's understand this

    Following our chart, these are the steps followed during development of software in an organization.
    1. User requirement
    2. System requirement
    3. Global design
    4. Detailed design
    5. Implementation
    First step is to get user requirement.
    Then we develop a system requirement.
    Then comes the global design.
    Next is detailed design.
    And the last step is implementation, where software is developed.

    When we say work product in the system requirement stage, the output is system requirement whereas the global design and detailed design stage output is design document. And in the implementation stage, output is code. So keep in mind, work product means output.

    Test item, which is also known as test object, is any document or component or system which is under test.

    Example:

    In the requirement stage, requirement is work product. If this has to be reviewed, then requirement is referred to as test item. Similarly, if we are in implementation stage, output of this stage is code. And if we want to perform testing on it, then code is our test item here. So output of the stage is called work product. And if testing is performed on it, then it is called test item. So we can say any document or component or system which is under test is called test item.



    This is the first objective to evaluate work products such as requirements, user stories, design and code.

    We already know work product means output and output of the requirements stage is requirement document where all the requirements are mentioned. Now, as per the first objective, we have to evaluate this work product.

    Example:

    Suppose this is the customer requirement for Web page.

    Code:
    Requirement:
    For web page, when the login details are given the next page shall load in a few ms and if login details are not correct then show a popup.

    Summary:

    When the login details are given, the next page shall load in few milliseconds. And if logging details are not correct, then show a pop up.

    But if you look carefully, this requirement is not complete. There are open points. First, how much time? Second, which page will load next? Third, what is the pop up content? These are the questions which need clarification. So this is why it is necessary to evaluate work products such as requirements, user stories, design and code.



    Second, objective states to verify whether all specified requirements have been fulfilled. 

    Let's understand this:

    We already know this requirement. Now we need to see if these requirements are fulfilled in the engineering requirement. After analyzing the customer requirement and asking question, we finally have the engineering requirement, which includes the following points. First, Log-in is correct. Go to next page. Second next page shall load in five hundred milliseconds. Third next page shall contain the personal information. Fourth, if the lock in detail is not correct, pop up shall appear. Five pop up message, password or user I.D. is incorrect.

    Like this, in each stage, we have to verify whether all specified requirements have been fulfilled.



    Third objective is to validate whether the test object is complete and works as the users and other stakeholders expectations.

    Test object means object under test. And it is defined as the component or system to be tested when you are in requirements. Stage requirement. Is your test object. And in design, stage design is your test object. And at the implementation stage code is your test object.

    Example:

    As per third objective, we need to provide input to the test object and check the output if it fulfils the stakeholders requirement. That means once the software is ready, we need to executed to see if it's fulfilling the customer's requirement. Since here we are executing the code. So this comes under validation. So third objective is to validate whether the test object is complete and works as the users and other stakeholders expect.



    Fourth objective is to build confidence in the level of quality of the test object.

    Let's understand this:

    Suppose we are in the requirements stage, then requirement is our test object. And if we clarify our requirement in this stage itself, instead of clarifying it during the implementation stage, then we can build confidence in our requirement. And finally in our product. That is why fourth objective is to build confidence in the level of quality of the test object.



    Now let's discuss objectives five and six together:

    Objective five is to prevent defects and objective six is defined failure and defects.

    Let's understand these two objectives:

    If you find defects in the requirements stage, you prevent these defects to go to the next stage.

    Now, the sixth objective comes into picture.

    We need to find the defects or failure in the same stage in which they are tested. Otherwise, defect will travel to the next stage and it will be more costly to fix. This is one of the important objectives of testing to prevent defects. To find failure and defects.



    The seventh objective is to provide sufficient information to stakeholders to allow them to make informed decisions, especially regarding the level of quality of the test object. When we find defects, it's not necessary that we will fix all of it before release. But what we can do is to provide sufficient information to the stakeholders regarding defects and risks associated with it.



    The eighth objective is to reduce the level of risks of inadequate software quality, for example, previously undetected failures occurring in operation.

    This objective is related to the five and six objective.
    We need to find a defect in the same stage in which they are introduced.
    Otherwise, it will be found in the operation.
    If fault is found during operation, it can have adverse effect.
    That's why it is necessary to reduce the level of risks of an adequate software quality.



    The ninth objective is to comply with contractual, legal or regulatory requirements or standards and to verify the tests object's compliance with such requirements or standards.
    Sometimes you need to fulfill the legal requirements, for an example, if you are working for automotive industry, then you need to fulfill. ISO26262 standard for safety critical requirement.



    Till now, we discussed general testing objectives, whereas the objectives are context dependent. To understand this, let's have a look into two different testing levels.
    Component level and acceptance level. When you perform testing at component level, your objective is to find as many defects as possible so that they are not found during operational use. Whereas if you aren't an acceptance level, your objective is to check if system works as expected. So in different level of testing, the objective changes.

    Let's have a look at the important points:

    1. To evaluate work products such as requirements, user stories, design and code to verify whether all specified requirements have been fulfilled.
    2. To validate whether the test object is complete and works as the users and other stakeholders expect to build confidence in the level of quality of the test object.
    3. To prevent defects.
    4. To find failures and defects.
    5. To provide sufficient information to stakeholders. To allow them to make informed decisions, especially regarding the level of quality of the test object.
    6. To reduce the level of risk of inadequate software quality, for example, previously undetected failures occurring in operation.
    7. To comply with contractual, legal or regulatory requirements or standards, and to verify the test object's compliance with such requirements or standards.
    8. The objective of testing is context dependent.




    First, we will have a look into testing activities:

    Most important activity of testing is to show failure, because we know by now one of the objectives of testing is to find defects. Finding defects is not the only activitie of testing, other activity of testing is once the defect is found. It is very much required to check if found defect is fixed or not. And the last point is testing is done by testers.

    Now, let's have a look into the debugging activities:

    Most important activity of debugging is to analyze failure.
    Once the defect is analyzed and root cause is found, next activity is to fix the defect.
    And the last point is debugging is done by developers.

    Now here we will discuss how Tester and developer work with respect to defect cycle.
    1. First Tester finds the defects, then the found defect is reported to the development team.
    2. After getting the defect report, development team starts investigating the failure.
    3. While investigating the failure, developer isolates the defect from rest of the software.
    4. Once the defect is isolated, developer fixes the defect and then checks defect is fixed or not.
    5. Once the defect is fixed, the fixed report is sent to the testing team.
    6. After getting the fixed report from developer Testor Retests on found defect to confirm if they were really fixed.
    One more important point to remember is in agile development and in some other life cycles, testers may be involved in debugging and component testing, though in general, debugging is developers task.
    But in agile development model, sometimes debugging is done by testers.


    Let's have a look into difference between testing and debugging:
    1. Very first difference is the testing is performed by Tester and debugging is performed by developer.
    2. Second difference is testing finds the programming failure, whereas debugging is to demonstrate that program is working fine.
    3. Less difference is testing is done with the purpose of finding bug, whereas debugging is done to find the cause of the bug.
    Let's have a look into the important points:

    In agile development and in some other life cycles, testers may be involved in debugging and component testing.

    Testing activities are finding defects and confirming defects are fixed, whereas developers activity is to analyze defects and fix the defects.

Which means if there is some wrong or incorrect in the requirement and we don’t spot it then that is how the entire project will be developed. This is what we have to stop by performing testing on requirement.

    Example:

    Let’s have a look unto the software development levels in ascending order:

    1. User requirement
    2. System requirement
    3. Global design
    4. Detailed design
    5. Implementation
    If we do not correct the error at the system requirement level it becomes part of the global design and the detailed design, both of which will be wrong and eventually we will have a software that doesn’t work correctly.

    So that is why if we correct the requirement at this stage the rest of the process will carry out without any errors due to requirement. This is why testing the requirements contributes to the success of product.



    At design level there will be a system design and our job is to do a review on it. Since there is no code so we can’t run it. Therefore we review it which is a static testing technique where we do this review it will increase team members understanding. So whether is the tester, developer or the process guys everyone will have the same understanding after review. If everyone has the same knowledge it will help reduce the risk of fundamental design defects. So a review of the system design will reduce the risk of fundamental design defects.

    Example:
    1. User requirement
    2. System requirement
    3. Global design
    4. Detailed design
    5. Implementation
    We are now we are at the global and detailed design stage. If we carry out the testing here and catch the mistake then it will not go on to the implementation stage. We will send the correct software for the dynamic testing and product will be a success. If we don’t do the testing then we will send a faulty software for dynamic testing and then we will be forced to go back to stepts to find the error.

    Conclusion is if we perform testing at the design level it contributes to the success of the whole project.



    In the coding level you have code under development. The code is being developed and you will perform the static testing on it. The intention with this is to increase the understanding of that piece of code. What this does is reduce the risk of defects within the code. So if there are any defects we can find them out by reviewing the code through static testing.

    Example:

    Customer requirement - Check that the window will move up with in 10ms once the button is pressed.

    Code:
    UpMovementFunc()
      {
        if(PressTimer <= 10)
        {
          WindowMovement = Up;
        }
    }


    The developer has written if the button is pressed for 10 milliseconds or less then the window will move up. The requirement was for the window to move up within 10 milliseconds not at 10 milliseconds. The developer has included the less < operator which changes the result, now the window might not move up unless exactly 10 milliseconds have passed.

    This is how performing testing at the coding level can reveal mistakes in the code and contribute to the success of the whole project.



    Once the software is ready we verify and validate the software. Now that the software is functional and the codes are ready we can perform dynamic testing on it. Once we perform dynamic testing we detect failures that might otherwise have been missed. If we don’t do the testing at this level then there is a fair chance that we will miss some errors and they will finally be discovered when the software is in operational use. Next is removing the defects that cause the failures so we will fix the problems because of which the software is failing. This is where the debugging process comes into play. The tester finds the error and tells the developer who fixes the problem and sends it back to the tester to confirm that the error is gone. And by doing all of this our software will meet the stakeholders needs, it will meet the customers or the users requirements and that means everyone using the software will be happy.

    So lets understand the relationship between quality assurance and testing.

    Here we have two concepts:



    Quality assurance is also called Q.A.



    People often think that quality assurance and testing are the same. That is a mistake. They are not the same, but they are related. They are related through a larger concept known as quality management. This concept ties them together. So you have to remember that. While quality assurance and testing are not the same, they are tied together by a larger concept called quality management.

    Explanation in a nutshell:

    First step in an organization, quality management is established. Second, we have quality assurance as a subset to ensure that the quality requirement will be fulfilled with the help of standards and procedures. On the other hand Q.C. (Quality Control) is to check if quality is fulfilled or not. This checking is done using different testing techniques.



    Q.M. is a series of coordinated activities to direct and control an organization with regard to quality. So we are establishing a management system in an organization, and the objective of this management system is to achieve the quality requirement of the organization.

    So what is quality management? It is a larger system introduced into an organization with the single goal of achieving the best quality.



    This is a part of quality management, quality management is a bigger system, and out of that, we get quality assurance. And the purpose of Q.A. is to focus on providing confidence that quality requirements will be fulfilled.



    We already have a management and we have a standard and procedure. Now we need to implement these concepts. This implementation is done by using quality control. It is the operational techniques and activities, part of quality management that are focused on fulfilling quality requirements.

    Overall:
    • Quality management is the larger system, which has two parts quality assurance and quality control.
    • Quality assurance contains all the documents that tell us what procedures to follow and quality control is, the activities that we have to perform in order to achieve the organization's quality requirements.
    • Quality control is an activity, quality assurance is a document or a process, and quality management is the complete system under which quality assurance and quality control work.


    So quality assurance contains proper processes while quality control has test activities. So Quality Control has activities that are practical, whereas quality assurance is a theoretical processes. The next point to look at is quality assurance is more about defect prevention by laying out guidelines and documents. This helps prevent defects. The quality control is a defect detection mechanism.

    Recap:
    1. First point is quality management type quality assurance and testing together.
    2. Point two is quality assurance is not the same as testing.
    3. Third point is definition of quality. Quality is conformance to a requirement that means in software testing. If we talk about quality, it means we are meeting the requirement.
    4. Fourth, an important point. Testing is part of quality assurance.
    5. Fifth is quality control is an end of phase activity. If you are in the requirements stage, once the requirement is written, the requirement undergoes testing process as a part of quality control. Similarly, for other stages. Once the activity of that stage is done, quality control comes into picture. That's why we say quality control is an end of phase activity.

    Let's start with the definitions, and afterwards I will provide you the detailed explanation.



    An error is a human action that produces an incorrect result.

    To relate this definition to software testing let's say, if the developer finds the mistake in their own code, then it is referred to as an error.



    All of them is a flaw in a component or system that can cause the component or system to fail to perform its required function.
    Now, that's related to the software testing, once the software finds the bug and developer accepts it, then it is called as fault.



    A failure is a deviation of the component or system from its expected delivery service or result.

    Suppose you are using the software, but it hangs. That means if the product is in use and it's not working as expected, then it's a failure.

    Now, let's understand each of these definitions in more detail.

    I - If a human makes a mistake in writing a code, then we call it an error.
    II - If the defect error found by a tester is called defect.
    III - When this defect is accepted by the development team, then it is called a bug.

    I - If a programmer finds a mistake in their own code, then it is an error.
    II - If the mistake is found by the Tester, then it is a defect.
    III - If that mistake is confirmed to be a problem, then it is called a bug. And when the bug is in this stage, then it is called a fault because it is the cost of a failure.



    Once the system is built. But it is not meeting the requirements, then it is a failure. So what is causing the failure? The mistake which was not found during fault stage?

    Now, let's use an example to understand this better.

    Say this is a requirement

    Code:
    IF speed is 120 km/hr or more "over-speed" warning shall come

    Now, say this is a code written by a developer, if speed is greater than 120 kilometers per hour. If he finds the mistake here, then it is an error.

    What is the mistake here? The requirement is if speed is one hundred and twenty kilometers per hour or more. But here it is only greater than 120 kilometers per hour. So when speed is equal to one hundred and twenty kilometers per hour, the overspeed warning will not be shown. If he notices the mistake here, then it will be called an error. If this mistake is found by a tester, then it is a defect. If the developer confirms that, yes, you're right, it's a mistake. Then it will be called a bug. And normally developer finds out how many bugs are there. While Tester just says that there is a defect.

    This is how these terminologies are used in an organization, and this stage is called a fault.

    Now, let's say that the mistake is not found in the development stage and the code becomes a part of the car. So now if we find this mistake, then it is called a failure. Before we end this video, let's have a look into the important points now, you shall be in a position to differentiate between fault and failure. Fault is found by the tester and in the development environment, whereas failure is found by the user in the operational use. And it happens due to deviation from the requirement. And the last point is fault is the cause of the failure.



    There are two main focuses of this discussion.

    Normal causes of defect and environmental causes of defect.

    * Normal defect costs *

    Time Pressure

    If you are working in an environment where you are given very little time to complete your tasks, then it is possible that you will overlook certain things that may cause a defect.

    Human fallibility

    As humans, we are all fallible because fallible means likely to make errors or fail. Nobody's perfect. After all.

    Inexperienced and insufficiently skilled people

    If you work in an organization where there are people without sufficient knowledge of the product, then it may result in a defect.

    Miscommunication or misunderstanding

    If your organization lacks proper channels of communication, this could also lead to defects.

    The complexity of the code, design and architecture

    This means even if you are an experienced, skilled person, if the code is complex, then you might end up making an error.

    New, unfamiliar technologies

    If you are working with a technology that you don't know well enough, then this can also result in a defect.

    So these words that, in fact, causes.

    Radiation

    Radiation can cause a defect.

    Electromagnetic fields

    We all know how on flights we are asked to switch off our mobile phones. This is to avoid creating electromagnetic fields which can cause interference.

    Pollution

    So if there are dust particles on the sensor, it could result in an error.

    And similarly, there could be many other environmental causes.

    It is the earliest actions or conditions that contributed to creating the defects. This means that when you find a defect, you also have to find the first condition that caused the defect.

    Example:
    Code:
    Requirement:
    Once the speed is more than 115, red light shall glow

    After getting this feature, customer is unhappy because customer check this device and made the observation that there is a defect. Because he said, when the speed reached to 116, he expected the red light to be on. But he found that it's still off.

    Now, to figure out why this is a defect, we have to do a root cause analysis. We are in the testing stage now, but we have to go back a step to the implementation stage or coding stage. Here we find that there is a condition that was implemented incorrectly, if speed is equal to or greater than one hundred and fifty.

    However, our job is not yet done. Remember, we have to find the earliest condition. This means that we have to go one stage back again, which means the requirement or design stage in this stage. We found that some of the developers have written this system requirement.

    Code:
    Requiremet:
    Red light show glow when speed is more than 150


    So instead of one hundred and fifteen, they rode 150. And because of this, the implementation was done incorrectly. And we found the defect in the coding. However, our job is not yet done since we have to find the earliest condition. When we investigated more, it was found that since the communication was verbal, this problem occurred. 

    Now, your job is to identify fact, failure, fault or root cause and customer is unhappy is a fact. Observation made by customer is failure. These two statements are defect, and all of this problem occurs due to miscommunication. So this is the root cause.

    But we don't stop at this point. Now we need to find a solution to avoid such problems. So action over here is further communication will be done via email. We will see one more example to illustrate what we learned about defect and root cause analysis.

    Example:

    Here is the life cycle of diffract root cause analysis:

    This life cycle starts with a customer complaint, which is also the fact.

    Why is he complaining? Because he has come across a failure. He has received incorrect interest payment calculation. So the customer is trying to calculate his interests, but received an incorrect result. This was a failure.

    When we analyze this failure, we find that it was caused by a single line of incorrect code. When we further investigate this defect, we find out that the wrong code was written because of the product owner's misunderstanding of how to calculate interest. This was the root cause of the defect that made the customer complain.

    So we have this information. The product owner didn't know how to calculate interest, but we can't stop there. We have an entire team. So how is it that the product owner could make this mistake? What we do next is called action. We figure out what to do, so this never happens again. We train the product owner and interest calculation. This way they will not repeat this mistake again. So you see how everything is connected. This is how root cause analysis works. It starts from an effect that is the complaint and ends with an action. That is our method to correct the root cause.

    Now we'll look at why we need to do a root cause analysis.

    1. The first point is to prevent a significant number of future defects from being introduced.
    2. We want to make sure that new defect don't crop up in the future.
    3. The second point is to reduce the arguments of similar defects in the future, so we don't want this interest calculation error to crop up again in our system.
    4. The third point is to improve the process.
