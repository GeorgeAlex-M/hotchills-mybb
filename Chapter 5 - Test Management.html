    Chapter 5

    Test Management

    5.1 Test Organization

    Independent testing - A certain degree of independence often makes the tester more effective at finding defects due to differences between the authors and the testers thoughts. 

    Suppose we have a tester and a developer. During independent testing, testers and developers are provided with requirement, and they analyzed the requirement independently. After analysis the developer implements the code and tester designs test cases. Since they develop their work product independently The probability of finding defect increases. This is the biggest advantage of independent testing. 

    Degree of independence - Is used to represent how independent the testing team is. 

    Example:
    - Developer testing their own code 

    Here, the developer who implements the code is testing their own developed code. Therefore, there is no independence in testing. 

    - Testers within the development team

    Here we have a team and it consists of both tester and developer. Now in this case, the degree of independence is more because the developer implements the code, but testing is not done by them. Testing is done by the tester. So we can say the degree of independence is more comparing to the last example. 

    - Independent test team, same organization, same manager

    Suppose you have an independent test team. This test team is not part of development team. But the test team is in the same organization and under same management. In this case, again, the degree of independence is more compared to last example as the test team is separated from the development team. 

    - Independent test team, same org, different manager

    Here the test team is independent and organization is same, but the managers are different. So here again, the degree of independence is more comparing to last example, because here the managers are different.

    - Independent test team, different organisation

    Here,the development team and testing team are in different organizations and all the previous examples, both the team were in same organization. And here they are in different organization. So it's very obvious that the degree of independence is more here. 

    Now, here we will see some of the benefits of independent testing:

    1. First benefit is that we can find different kinds of failures. 

    Independent testers are likely to recognize different kinds of failures compared to developers because of their different backgrounds and technical perspectives. 

    2. An independent tester can verify, challenge or disprove assumptions made by stakeholders during specification and implementation of the system.

    3. Independent testers of a vendor can report in an upright and objective manner about the system under test without pressure of the company that hired them. 

    There are a few drawbacks of independent testing also, which we will address here:

    1. Isolation from the development team

    It may lead to lack of collaboration, delays in providing feedback to the development team or an adversarial relationship with the development team.

    2. Developers may lose a sense of responsibility for quality

    3. Independent testers may be seen as a bottleneck

    For example, during the release, if Tester finds a critical defect, then sometimes people think due to Tester. We are not able to release on time. 

    4. Independent testers may lack some important information.



    Table arranged in the order of degree of independence. Highest degree is on the top.

    5.1.2 Tasks of a Test Manager and Tester

    Task of the test manager

    1. Develop or review a test policy and test strategy for the organization. 

    Remember one point, any task which can impact the overall test activity is done by the test manager. Since the test policy and test strategy has an impact on the overall testing, it is developed by the test manager. 

    2. Plan the test activities by considering the context and understanding the test objectives and risks 

    Here also, you can see the test plan has impact on the overall testing and therefore, this is also task of the test manager.

    To plan the test activity the test manager:

    - Selects test approaches
    - Estimates test time, effort and cost
    - Acquires resources
    - Defines test levels and test cycles
    - Plans defect management

    3. Write and update the test plan. 

    And we also know that the test planning is done for the complete testing activity. So this test is also done by the test manager. 

    4. Coordinate the test plan with project managers, product owners and others.

    That means overall test coordination is done by the test manager.

    5. Is responsible for shared testing perspectives with other project activities, such as integration planning. 

    6. Initiate the analysis, design, implementation and execution of tests, monitor test progress and results and check the status of exit criteria and facilitate test completion activities. 

    So that means the test manager initiates the testing task and tester work for that task. 

    7. Prepare and deliver test progress reports and test summary reports based on the information gathered. 

    Tester performs the testing activity and provides the data to the test manager. And using this data, test manager prepares and delivers test progress report and test summary reports. 

    8. Adapt planning based on test results and progress and take any actions necessary for test control. 

    As we know, testing is performed based on the test planning provided by the test manager. But after analyzing the data, if manager feels that the target is not achievable. Then is the task of the test manager to adapt the test plan. So the target can be achieved. 

    9. Test manager, also support in setting up the defect management system and adequate configuration management of testware. 

    - Defect management system records all the defect of the software. These records can be seen later as well if you want to review or want to check that how you have fixed that. These records can be used the knowledge-base for the future reference. 

    In simple terms, defect management system manages the defect found during the software development cycle. 

    - Configuration management is a form of I.T. service management that ensures the configuration of system resources, computer systems, servers and other assets are known, good and trusted.

    In simple terms configuration management is used to maintain the work product of the software development cycle. 

    10. Test manager also introduce suitable metrics for measuring test progress and evaluating the quality of testing and the product. 

    We know that during test, these metrics are generated, but which all data we need to collect is decided by the test manager. Afterwards this data is used for adapting test plan. 

    11. Test manager supports the selection and implementation of tools to support the test process, including recommending the budget for tool selection, allocating time and effort for pilot projects, and providing continuing support in the use of the tool. 

    Selecting of test tool will also impact the complete testing activity. And that's why this is task of the test manager. 

    12. Test manager also decide about the implementation of test environment. 

    With the help of the tester, test manager decides how and when to implement test environment.

    13. Along with project related task, it is also task of the test manager to promote and advocate the testers, the test team and the test profession within the organization. 

    14. It is the task of test manager to provide environment so that Tester can develop their skill.

    Tasks of a Tester

    1. Review the document and contribute to test plan. 

    Now you can see the difference. Task of the test manager is to make test plan and task of the tester is to implement the test plan. 

    2. Analyze, review and assess requirements, user stories and acceptance criteria, specifications and models for testability. 

    So Tests manager initiates the analysis process, but analysis is done by the tester. 

    3. After the analysis of the requirement, the tester has to identify and document the test condition and provide traceability. 

    4. We know that after test analysis, we have to do test design. Again, tests manager initiates the test design task, but is performed by the tester. Along with the design, the tester is involved in test setup and verifying test environment, often coordinating with system administration and network management. 

    5. Implement the test cases and test procedures. 

    6. Prepares and aquires test data as an input for the testing. 

    7. As we know, after test implementation comes the test execution and Tester is involved in that. 

    Tester creates the detailed test execution schedule, and we use this for test case execution. 

    8. Tester executes the test cases, once the test cases are executed. Tester evaluates the result. If Tester finds the deviation between the actual result and expected, then Tester documents it. 

    9. This point is related to the usage of tool. If you remember, the test manager selects the tool, but Tester uses these tools to facilitate the test process. 

    10. Tester is also involved in automating the manual test cases. Tester can also review tests developed by the others. 

    11. Evaluate non-functional characteristics such as performance efficiency, reliability, usability, security, compatibility and portability. 

    Role of the tester in different test levels

    1. Component testing 

    Usually done by the developer. 

    2.Acceptance testing

    It is performed by the business analyst.

    3. System test level

    It is done by the independent test team. 

    4. Operational acceptance testing

    It's performed by the system administration staff.

    5.2 Test Planning and Estimation

    Definition - Test plan outlines test activities for development and maintenance projects. 



    Test plan provides us roadmap to know when which activity will start and which activity will end. In test plan we decide when the testing will start, when test analysis and design will start, when will implement and execute the test cases, wWhen we will end our testing and when software will be released. But now the question, can we predict all the challenges before we start the project? The answer is no. We cannot. During testing, we may face issues and it may impact our initial planned activity. At that time, how we'll come on track. So to handle such situation, we have monitoring and control stage. We monitor each and every stage of testing. And if we see we are not going as per plan, monitoring stage gives feedback to the test planning stage and based on the feedback, the preplanning is done. So this is how planning process works. So now my question to you is, is test planning a one time activity? The answer is no. Test planning is a continuous activity. This point is very important. Remember it.

    Different factors that can influence test planning:

    1. Test policy and test strategy

    2. Development lifecycle

    Like V model or agile model

    3. Third is scope of testing

    For example, you only want to do component testing or integration testing or system testing. 

    4. Objective and risk

    As we know, objective of system testing is different from objective of acceptance testing. In system testing we tried to find as much bugs as possible, but in acceptace testing our objective is to see that there are no bugs. 

    5. Constraints and criticality

    As we said before, testing of mobile phone will be done differently than testing of an aircraft.

    6. Testability and availability of resources

    This is quite clear. If you have 10 people in team, you will plan differently than if you have 6 people in team. 

    Test Planning Activities

    1. Determining scope, objectives and risk

    In test plan we define the scope of testing. Then we determine the objective of testing. And we also list down the risks associated with the object under test. 

    2. Defining the overall approach of testing



    One example which I can give you here is model basis testing approach. 

    3. Integrating and coordinating the test activities into the software lifecycle activities

    Since testing is not done in isolation, in test plan, we synchronize testing activity with the development activity. And what we focus here is when and which development activity will finish. So that we can perform testing on that stage. If we take example of V model, when the implementation will be done so that we can start component testing. So in test plan, we integrate and coordinate the test activities into the software lifecycle activities. 

    4. During test plan we determine what to test

    To do this test, we have to figure out how many resource we need. And we also determine how test activities will be carried out. 

    5. Scheduling of test activities

    In test plan we scheduled these test activities. We already saw this where we schedule when the test will start, when these activities will be performed. And finally when testing will end. So scheduling is part of test planning. 

    6. Selecting metrics for test monitoring and control 

    As we know, monitoring and control phase gives data to the planning phase. But what type of data we need to collect in monitoring stage is determined in the planning stage. So remember this point. Selecting metrics for test monitoring and control is done in planning stage. 

    7. Budgeting for the test activities 

    This is done in planning stage. 

    8. Determining the level of detail and structure for test documentation

    We determine the structure for the test documentation. 

    If you want to know more about test planning, you can refer to ISO/IEC/IEEE 29119-3 standard.

    5.2.2 Test Strategy and Test Approach

    Definition - A test strategy provides a generalized description of the test process, usually at the product or organizational level. 

    In simple terms, a test strategy defines which type of approach we will follow for testing. 



    Tests process, generalized description of the test process is provided by test strategy and then which tests strategy to use depends on test approach, complexity and goal of the project, type of product being developed and product risk. And which test approach to use is based on risks, safety, available resources and skills, technology, the nature of the system, test objectives and regulations. Now, based on the test approach, we select test techniques, test level and tests type. 

    So this is how they are connected. At the top we have a test process, input to this is coming from the test strategy and input to the test strategy comes from the test approach. Test approach also helps in selecting test technique and test types. 

    Test strategies

    1. Analytical strategy

    As the name says, this type of test strategy is based on an analysis of some factor. These factors can be a requirement or risk. In simple terms, if you have to analyze requirements for testing, then it will come under this strategy. Similarly, if you have to test based on risk analysis, it will come under analytical testing. 

    So risk based testing is an example of an analytical approach where the test is designed and prioritized based on the level of risk. Analyzing requirement and risk is an example of an analytical strategy. 

    2. Model based strategy 

    Here, tests are designed based on some model of some required aspect of the product, such as a function, a business process, an internal structure or a non-functional characteristic. 

    The model based strategy is based on function, a business process, an internal structure or a non-functional characteristic.  Example of these models are a business process model, state model, reliability, growth model. 

    3. Methodical strategy 

    Here, test strategy relies on making systematic use of some predefined set of tests or test conditions. Here you will have existing data like predefined tests or test conditions, and you will use this for your testing. 

    Now, let's see an example of it. First is taxonomy of common or likely types of failures. Here you can have a list of failures already existing from your previous project and now you will use it for testing. Second example is a list of important quality characteristics. Here you have some standards where the important quality characteristic is already listed. Now for your project, you will use this predefined list from the standard. And the last example is company wide look and fuel standards for mobile or web pages. 

    4. Process compliant (or standard compliant)

    It is also called a standard compliant. This type of test strategy involves analyzing, designing and implementing tests based on external rules and standard. Don't get confused with the word analyze and link it to the analytical model. In the analytical model we analyze the requirement or risk of the product, but in process compliance model we analyze, design and implement tests based on external rules and standards. When we say external standard, that includes industry specific standards, process documentation, the rigorous identification and use of the test basis, any process or standard imposed on or by the organization. 

    5. Directed (or consultative)

    Is sometimes also referred to as consultative. This type of test strategy is driven primarily by the advice, guidance or instructions of stakeholders, business domain experts or technology experts who may be outside the test team or outside the organization itself. 

    6. Regression-averse

    This type of test strategy is motivated by a desire to avoid the regression of existing capabilities. Here we tried to use the existing things available from past project, so we don't spend time on it again. 

    Let's see what all things we can reuse. We can reuse existing test environment, test cases and test data, extensive automation of regression tests, standard test suits. Just remember, in regression averse test strategy, we try to use the existing infrastructure. 

    7. Reactive 

    In this type of test strategy, testing is reactive to the component or system being tested and the events occurring during test execution rather than being preplanned. Here we plan during test executing, for example, if lots of defects are logged, then we plan accordingly. So here planning is done during test execution. To do this, we need an experienced tester who can design and implement a test case and immediately execute it based on the result from the previous test result. For example, in the case of exploratory testing. 

    The last point with respect to test strategy is test strategy is often created by combining different test strategies. For example, we can combine risk based testing with exploratory testing because they complement each other and many achieve more effective testing when used together. So the point to make is test strategies can be combined.

    5.2.3 Entry Criteria and Exit Criteria (Definition of Ready and Definition of Done)

    Definition - Entry criteria are the preconditions for undertaking a given test activity. 

    Let's understand this. Suppose there is a testing phase. This could be any testing phase. Test design, test implementation or test execution. You want to start this testing phase, but before you could start, you have to look into the preconditions. Once all the preconditions are met, only then you can start this test activity. This is to ensure that you have everything that is required for testing. Similarly, if you want to end this testing activity, then you cannot suddenly end it. You have to see if all the post conditions are met. And once you find they are all met, you can stop the test activity. So here, preconditions are called entry criteria and post conditions are called exit criteria. 

    Definition - Exit criteria define what condition must be achieved in order to declare a test level or a set of tests completed

    Remember, entry criteria is referred as definition of ready and exit criteria is referred as definition of done. 

    Examples of entry criteria:

    - Availability of testable requirements, user stories and/or models. (For example, when following a model based testing strategy.)
    - Availability of test items that have met the exit criteria for any prior test levels
    - Availability of test environment
    - Availability of necessary test tools
    - Availability of test data and other necessary resources

    Examples of exit criteria:

    - Planned tests have been executed
    - A defined level of coverage (For example, of requirements, user stories, acceptance criteria, risks or Code has been achieved.)
    - The number of unresolved defects is within an agreed limit. 
    - The number of estimated remaining defects is sufficiently low.
    - The evaluated levels of reliability, performance efficiency, usability, security and other relevant quality characteristics are sufficient.

    5.2.4 Test Execution Schedule

    We will address less theory and more practical based questions here. Let's first understand the scheduling concept and then, based on this understanding, will solve some questions. 

    So as we know in testing, first we analyze requirement. Then we write test cases or test procedures. Now, these test cases or procedures can be for manual testing or automation testing. In manual testing we follow the actions mentioned in the test procedure to perform testing and for automation testing we develop scripts based on the test procedure. So now whether you perform manual testing or automation testing, you have to decide which test procedure or test script will be used first for testing and which one will be used at the end. And to take this decision, test scheduling comes into the picture. Because we have so many test procedures and test scripts, there shall be an effective mechanism to execute them. So we go for prioritization of the test cases. Prioritization can be based on risk or based on some other factors. But the main point is high priority cases are executed first. 



    Suppose these were the test scripts and these were their priority one being the highest priority. Then this is the order in which we have to execute the test cases. Now, let's use this concept to solve some practical examples.

    5.2.5 Factors Influencing the Test Effort

    Definition - Test effort estimation involves predicting the amount of test-related work that will be needed in order to meet the objectives of the testing for a particular project, release or iteration. 

    In simple terms here, you calculate how much time you need to complete the testing activity. And there are some factors on which test estimation depends, like characteristics of the product, characteristics of the development process, characteristics of the people and the test results. 

    Characteristics of each factor

    1. Product characteristics 

    Examples are the risks associated with the product:

    - the quality of the test basis
    - the size of the product
    - the complexity of the product domain
    - the requirements for quality characteristics, for example security or reliability
    - the required level of detail for test documentation
    - requirements for legal and regulatory compliance

    So all the things related to the product come under this. Like risk, quality, size, complexity, documentation and compliance of the product. 

    2. Development process characteristics

    All the inputs you need to develop the test process will come here like:

    - the stability and maturity of the organization
    - the development model in use
    - the test approaches
    - the tools used
    - the test processes
    - time pressure

    So these are the things we need for the development of the process, model, approach and tools.

    3. People characteristics

    - the skills and experience of the people involved, especially with similar projects and products
    - domain knowledge
    - team cohesion and leadership

    4. Test results

    - the number and severity of defects found
    - the amount of rework required

    5.2.6 Test Estimation Techniques

    Definition - In the metrics based technique, we estimate the test effort based on metrics of former similar projects or based on typical values

    Definition - In the expert based technique, we estimate the test effort based on the experience of the owners of the testing tasks or by experts.

    So it is clear in metric based technique, we use data from the previous similar project and in expert based technique, we use experience. 



    Now, let's see the example of metric based technique and expert based technique with respect to the agile model and sequential model. This table is enough to cover all the points. Burndown charts is an example of metric based technique under the agile model. And planning poker is an example of expert based technique under the agile model. Defect removal model is an example of metric based technique under the sequential model. And Wideband delphi is an example of expert based technique under the sequential model.

    5.3 Test Monitoring and Control

    Test Monitoring - The purpose of test monitoring is to gather information and provide feedback and visibility about test activities

    And we can gather information manually or automatically. 

    The gathered information is then used:

    - to evaluate the progress of testing
    - to check if exit criteria are met or not
    - to verify coverage target is met or not

    Test control - Describes any guiding or corrective actions taken as a result of information and metrics gathered and (possibly) reported.

    In simple terms, we gather information in the test monitoring stage and then we use the information to take corrective actions. 

    Now, let's see some of the corrective action examples:

    - Re-prioritizing tests when an identified risk occurs (for example, software delivered late)
    - Changing the test schedule due to availability or unavailability of a test environment or other resources
    - Re-evaluating whether a test item meets an entry or exit criteria due to rework
