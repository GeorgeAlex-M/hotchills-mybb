    The first point is testing can show that defects are present. So when we carry out testing and find a defect, then we know that there is a problem with the software. So testing can show that defects are present.



    Just because you have not found a defect it doesn’t mean that there are no defects in the software. So testing can't prove the absence of defects. Also, testing reduces the probability of undiscovered defects for example:

    Suppose to have 20 defects in a software. And when you perform testing, you find five defects out of it. So you have 20 defects and you found five. So the remaining defects are 15, which could not be found. So this is why we say that testing can show that defects are present. We found five. So we know for sure that there are five defects in the software. But we cannot prove that there are no defects in the software. We found five out of 20 defects. But there are fifteen left that we don't know about. So we cannot prove that there are no defects left in the software.



    Let’s continue with our example:

    Initially, we had 20 defects, but we found five. So only fifteen are left. We have reduced the number of defects that the customer can find.



    The first point to understand about this is testing everything is not feasible just because we have a software doesn't mean we can test every single bit of it.

    The second point is, whenever we do testing, it has to depend on risk analysis. What is the risk of that particular system? Depending upon that, we have to see what kind of testing is required. Then we focus on what types of testing are needed and what are the priorities of different features. We can't just test everything. We need to prioritize certain parts based on how important they are.

    The final point is to focus test effort. When we are testing a feature, we need to know how much effort it requires. This is why exhaustive testing is impossible. Testing requires a certain amount of effort, and that is why testing everything is not feasible.

    Example:

    Customer requirement:

    Code:
    LCD should glow when speed is between 15 and 90 kilometers per hour.

    So there is a lead that will only glow when the speed is between 15 and 90. One way of testing this requirement is to do a boundary value analysis for this test. We have taken six values on one end. We have 14, 15 and 16. And on the other end, we have 89, 90 and 91. So when we do the testing, the led should not glow at 14 or 15. And on the other, it should not glow for 90 or 91. It should only work for 16 and 89. So only by using six values we can test if the software meets the customer's requirement. But it is also possible to test with all the values between 15 to 90, but that will consume a lot of time and effort. Also, the boundary values serve our purpose. So this is the reason that we say that testing each and every part of the feature is not feasible. It is possible, but not practical. Of course, if the customer asks us, then we can perform testing on all the values in between. 



    Static and dynamic test activities should be started as early as possible.

    It means that we should start testing as early as possible, whether it is a static test or a dynamic test.

    Early testing is sometimes referred to as shift left. The very first stage of testing is that the left most corner of the software development lifecycle so early testing is sometimes called shift left.

    It helps to reduce or eliminate costly changes.



    So here is a very popular graph about the costs and time of finding defects. As you can see in the requirements stage, the cost of locating a defect is the lowest when you move to the design stage. It becomes a little costlier. Then as you move to the build stage and then to the test stage, it gets more and more expensive. Finally, at live use, it is the costliest to discover a defect in the product. So why does this happen? To understand this, lets look at this waterfall diagram model.



    Suppose we find a defect in the implementation stage, which is the build stage. Just because it was found at this stage does not mean that the error happened here. It might be in the design or the system requirement. It can also be that someone misunderstood the user's requirements because of which every subsequent stage was wrong and we implemented it incorrectly. So now to correct this one defect, we have to go back and correct this document and the others. All of this will take a lot of time, labor and resources. Which means that it will cost a lot of money. This is why Principal Three says perform testing as early as possible. If we find an error here, then we can correct it here and every other stage will be fine.



    Sometimes a small number of modules can contain most of the defects. Those defects can be responsible for most of the operational failures. Though these are only a few modules, the defects present in them can cause operational failure. What is operational failure? It is when the product is in use and fails to perform according to expectations. So if we can identify the module or the part of the software that contains these defects, then it will help us reduce or eliminate costly changes. If we find them during our testings, then they won't crop up during live use. And we can save the expenses on correcting them. So that's the fourth principle to identify the module where most of the defects are clustered together.



    The same tests no longer finds defects during testing. If you've already found a defect in a code, then it is unlikely that you will find a defect again unless there is a big change in the code. You will find no more errors since you already found one and corrected it. 

    To detect your defects existing tests and test data may need to change. So you either have to update the test case or the data in it to be able to find new defects in it.

    When you're performing automated regression testing, the pesticide paradox has a beneficial outcome. When you run automated testings, if you run the same script it is highly unlikely that you will find new defects. But if you update your script, add new values to it or new test cases to it. Then there is a possibility of discovering new defects. So this paradox is very beneficial when you're performing an automated testing.

    Suppose there is a software with 20 defects in it when you run your script the first time you find five defects. Then the software is left with 15 defects. Now, if you don't change the test script and you run it for the second and third time, you will find no more defects in the software no matter how many times you run it. You cannot find the 15 remaining defects. This is because you're running the same script and in order to find new defects you have to update your script. So this is why the same test will no longer find defects. To detect a new defect, existing tests may need to change.



    This means that the type of testing will depend upon the kind of product being tested. This is why testing is context dependent. So let's start breaking this down. Safety critical industry control software is tested differently from an e-commerce mobile app. So a mobile application and a safety critical software will be tested differently. We can't use the same process for both applications. Testing in an Agile project is done differently than testing in a sequential lifecycle project.
    Our methods of testing in an Agile project is completely different from how we test in a sequential lifecycle project. E-commerce is completely different and this is why Principle Six emphasizes that testing is context dependent.



    If you remember, we have already covered the essence of this in principle one and two. Let's deconstruct that.

    Point one in an organization. It is expected that testers can run all possible tests and find all possible defects. This expectation is completely wrong. We can't front all possible tests and we cannot find all possible defects.

    Point two says principal two and one, respectively, tell us that this is impossible. If you recall, Principle two says that exhaustive testing is impossible. And one says that we can never find all the bugs. We can claim to have found a bug, but we cannot claim that there are no defects left in the software. Further, it is a fallacy that is a mistaken belief. So this is an incorrect belief that a tester can find all the mistakes that exist in a software.

    Our final point here is thoroughly testing all specify requirements and fixing all defects found could still produce a system that is difficult to use. Even if we do everything, there can still be environmental conditions, unfound defects or other factors that can cause the product to fail. It's not in your hands. This is why we cannot say that there are no errors left in the software. If someone claims that, then it is a mistaken belief.

    As of now, we know there are seven test objectives. And to fulfill these test objectives, each one of us will have different ways. If we all apply our own ways in an organization. Then it is difficult to control. And it is less likely to achieve test objectives. That is a reason we need common process, which everyone shall follow as a guideline to achieve different test objectives.

    Seven common test activities can be applied for any product. These seven activities are: test planning, test monitoring and control test analysis, test design, test implementation, test execution and test completion. Altogether these activities are referred to as test process.

    Before we go to the test process topic you will have to understand the definitions for the following keywords:



    Definition - The body of knowledge, used as the basis for test analysis and design.

    In simple terms, test basis is nothing but the requirement.

    Example:

    Code:
    Requirement:
    We need a light system which will glow when door is open and close when door is closed. The system shall work like mentioned.

    The system shall work like mentioned, and this is a requirement which we need to test and it is defined as test basis.



    Definition - Is a testable aspect of a component or system identified as a basis for testing.

    Example:

    Code:
    Test Conditions:
    1. Light shall glow when door is open
    2. Light shall not glow when door is closed

    After reading the test basis we got this as a testable condition.



    Definition - Is a set of preconditions, inputs, actions (where applicable), expected results and post conditions developed based on test conditions.

    Test cases are sets of actions to test the test condition.

    Example:

    Code:
    Test Conditions:
    1. Light shall glow when door is open
    2. Light shall not glow when door is closed

    To check these test conditions, we can design two test cases:

    Code:
    Test Case 1:
    1. Switch on power
    2. Open door
    3. Check light ON
    4. Switch off power


    Code:
    Test Case 2:
    1. Switch on power
    2. Close door
    3. Check light Off
    4. Switch on power



    Definition - Is a sequence of test cases in execution order, and any associated actions that may be required to set up the initial preconditions and any wrap up activities post execution.

    This term is used mostly when you go for manual testing or you decide in which order test cases to be executed or if any precondition or post conditions are required.

    Now, let's have a look into test case 1:

    Code:
    Test Case 1:
    1. Switch on power
    2. Open door
    3. Check light ON
    4. Switch off power

    After the test case is executed, the door shall be closed and light to be switched off. Similarly, for test case 2, during initial condition, the door shall be open and light must be on so that we can see the change.



    Definition - Data needed for test execution.

    This term is used during the automation testing, while executing the test case we need test data.

    Let's see a simple data to understand this point:

    Code:
    Test Data ->

    Variable | Value

    Power Supply | 12V
    Door Status | Off:0 ; On:1
    Light | Off:1 ; On:0

    In the test case we are saying switch on power supply, but we didn't mention how much voltage shall be provided. This data we will get from Test Data. Similarly, all other values with respect to test cases are stored. These data can be stored in Excel form or any testing specific tools.



    Definition - Is a set of test scripts or test procedures to be executed in a specific test run.

    Test scripts means a set of instructions for the execution of a test.

    By using test case as a reference test, scripts are developed, these test scripts are then arranged in the execution order as shown:
    Code:
    SET-1 | SET-2

    test case - 2 | test case - 1
    test case - 4 | test case - 3
    test case - 11 | test case - 7

    That's why we can say a group of test scripts with a sequence of instructions is called suit.



    Definition - Work products produced during the test process for use in planning, designing, executing, evaluating and report on testing.

    As we know, we have different activities in test process, like test planning, test monitoring, test analysis, test design, test implementation, test execution and test closure. Each of these activities provide output and these outputs are called test wares. Test wares are stored for each testing phase for future reference.



    Definition - Is a source to determine an expected result to compare with the actual result of the system under test.

    As the definition says, test Oracle stores expected result, which is compared against the measured result during test execution.

    You should remember the following ISO standard: ISO/IEC/IEEE 29119-2 the standard defines vocabulary, processes, documentation, techniques, and a process assessment model for testing that can be used within any software development lifecycle



    Code:
    V-Model | Agile Model

    Incremental Process | Iterative Process


    V Model is an incremental process where all the phases are done sequently in an incremental way. Once the development phase is done, corresponding testing activity is started.



    In Agile model each phases are repeated throughout the development phase. Here few features are implemented in a week and testing is done for that feature and this repeats throughout the project development.




    Therefore, same test process are implemented in a different way based on the model used.



    Based on test level and test type, test process is selected.

    Here is a list of Test Levels and Test Types which we will discuss about later:

    Code:
    Test Level | Test Types

    Component Testing | Functional Testing
    Integration Testing | Non-functional Testing
    System testing | White-box Testing
    Acceptance testing | Change-related Testing


    Code:
    Products:

    Automotive | Avionics

    Ex: Cars | Planes

    Avionics product contains more risk than automotive. Therefore, the test process will be more regress for a product with more risk,


    Code:
    Software domains:

    E-commerse | Security

    Ex: Supermarket billing | Banking domain

    Clearly, supermarkets software will be more focused on load testing, whereas banking software will be tested more with respect to safety aspects.



    Example:

    Budget and resource - means if a signed budget is less or sufficient to complete the project.
    Time scales - means whether you need to complete a project in less time or you have sufficient time.
    Complexity - means if the product selected is complex to implement.
    Contractual and regulatory requirements - some times, along with the customer requirement we also need to fulfill industry related a requirement like for automotive safety. We need to fulfill ISO-26262 standard.



    The type of development model used has impact on the test planning and test process. Similarly, the type of test strategy tests, techniques and tools used in an organization influence the test process.



    As mentioned before ISO-26262 is a safety standard for automotive industry. It's an external standard which organization has to follow if they are working on automotive industry and need to be included in the test process.

    ASPICE is another standard which is related to process. If some organization needs to compliance to ASPICE standard, they have to adopt their process in accordance with ASPICE.

    There are three general aspects of organizational test processes.
    1. Test activity and task
    2. Test work product
    3. Traceability
    Test process is divided into seven different test activities and they are test planning, test monitoring and control test analysis, test design, test implementation, test execution and test completion. Each of these activities has task within them, which we will study in detail later. So when we perform these activities, we get certain outputs and those outputs are referred as test work product.
    Code:
    Test activities and tasks | Test work products

    Test planning -> Test plans
    Test monitoring and control -> Test progress reports
    Test analysis -> Prioritized test conditions
    Test design -> Test cases
    Test implementation -> Test suites
    Test execution -> Defect reports
    Test completion -> Finalized test-ware



    Suppose customer gave you some requirement to test. And the requirement looks like this:
    Code:
    Customer Requirement

    CR-1
    CR-2
    CR-3
    CR-4
    ..
    ..
    CR-n

    Once you get these requirements, you will develop test cases for these requirements. And suppose it looks like this:
    Code:
    Test Cases

    TC-1
    TC-2
    TC-3
    TC-4
    ..
    ..
    ..
    TC-9


    And it is up to you how many test cases you want to write to cover all the requirements. But now comes the important question. How will you provide the test case for customer requirement for? It is humanly not possible to provide test case for the requested customer requirement, if you have thousands of requirements. To solve this problem, we have traceability, during traceability we develop a virtual bidirectional link between requirement and test cases by using some tools. As soon as we ride the test case for requirement we link them using a tool. For example, requirement one is linked to test case one. Since we have a bidirectional traceability, anytime we can see which test cases connected to requirement one or which requirement is connected to test case one. Similar way we can provide traceability for other test cases, so that each requirement has test cases.

    Definition - To maintain traceability throughout the test process between each element of the test basis and the various test work products associated with that element.     Traceability is nothing but linking between requirement tests, condition, test, case and report.

    Test Process -> ISO/IEC/IEEE 29119-2
    Test Work Products -> ISO/IEC/IEEE 29119-3

    Test planning involves activities that define the objectives of testing. So test planning is the stage where we decide what we need to test and what we want to cheat from it.

    It also includes approach for meeting test objectives within constraints imposed by the context. We know that testing is context dependent, based on what we test, we decide a test planning stage and which approach we will use.

    Test plants may be revisited based on feedback from monitoring and control activities. The question is why we need to revisit the test plan? Suppose while testing you find that you cannot achieve the defined objective, it could be due to time constraints. In that case, we need to revisit test plan to see if we can increase the testing time or we increase resource to complete testing in same time, or we go for risk based testing, by keeping time and resource constant.

    So any time during testing, if we find that the defined objective or approach is not feasible, then we will need to revisit our test plan and update it.

    1. Scope

    - Determining the scope, objectives and risks of testing.

    In test planning stage, we determine the scope, objectives and risks of testing. Since test planning is the first activity of test process. We define the roadmap of testing here. We decide what will be the scope of our testing, whether we want to perform integration testing, software testing, system testing or all of this.

    Objective

    We can determine that if we want to do 50% coverage or we to perform testing on all priority features.

    Risk

    Once we get the project, we analyze the risk associated with the feature and based on that we decide whether to go for testing, based on the features implemented or based on analyze risk.

    2. Approach

    - Defining the overall approach of testing.

    We know that testing is context dependent. Based on what we test, we decide which approach we will use. For an example, we can go for risk based testing. If this release is critical to the customer, we can go for priority based testing to make sure that we perform the testing on critical features as early as possible so that if any bug is found, it can be fixed and released or go for requirement based testing to make sure if all the requirements are tested before release or choose related testing to make sure all the changed features are testing before release.

    3. Coordination

    - Integrating and coordinating the test activities into the software lifecycle activities.

    Traceability depends on the tool and to provide the traceability between all the work products, we need to integrate test activities into software lifecycle activities.

    4. What to Test

    - Making decisions about what to test, people and other resources required to perform various test activities, and how test activities will be carried out.

    5. Scheduling

    - Scheduling of test analysis, design, implementation, execution, and evaluation activities.

    This is the prepared roadmap during test planning.



    Here we define when testing will start, when test analysis and design shall be completed. Similarly, other test activities are planned.

    6. Metrics

    - Selecting metrics for test monitoring and control.

    In test planning stage we select metrics for test monitoring and control. Suppose in planning we design a matrix to activate 100% traceability. But if the testing team do not write test case for one of the customer requirements (in traceability), then we will not get the link. Since all the data is in tool by putting some filters, we can easily find out how many requirements are linked and how many are not. This way we get different data through Matrix.

    7. Budget

    - Budgeting for the test activities.

    In the planning stage we finalize budgeting for the test activities.

    8. Documentation

    - Determining the level of detail and structure for test documentation.

    Here we decide which document we need at the end of the test activity. And during testing we collect data for the documents.

    9. Continous Activity

    - Test planning is a continuous activity and is performed throughout the product's lifecycle.

    Test planning is not a one time activity, it is done throughout the development cycle.



    Definition - Test monitoring involves the ongoing comparison of actual progress against planned progress, using any test monitoring metrics defined in the test plan.

    Are supported by the evaluation of exit criteria, which are referred to as the definition of done in some software development lifecycle models.

    One of the important points why we do test monitoring and control is to communicate information to the stakeholder.

    As we know, test planning draws the road map for the test activity, which includes scheduling, like when testing will start and when it will end. Similarly for all other test activities, but in reality, it is not possible to provide the fixed timing and follow it due to many operational issues. That is the reason it is advised to monitor current status of each test activity against the planned one. So that if there is any lag, we can take necessary action to meet the plant's schedule.

    In test planning stage we decide when the analysis will start and when it will end. While monitoring, we compare the current status of test analysis with the plan schedule.

    Test Control

    Definition - It involves taking necessary actions to meet the objectives of the test plan.

    Evaluation of exit criteria

    Checking -> Test results and logs against specified coverage criteria.

    Assesing -> The level of component or system quality based on test results and logs.

    Determining -> If more tests are needed



    - During test analysis, the test bases is analyzed to identify testable features and define associated test conditions.

    Here we look for what to test also test basis is analyzed, as we know test basis is nothing but the requirement.
    Code:
    Requirement:
    We need a light system which will glow when door is open and close when door is closed.

    This is only a requirement, now from this requirement we need to find testable features. From this requirement, we get this as a testable feature:

    Code:
    Testable Feature:
    Light shall glow when door is open and close when door is closed.

    Next we have to find the test condition. From test feature, we finally derive these test conditions:
    Code:
    Test Conditions:
    1. Light shall glow when door is open.
    2. Light shall not glow when door is closed.

    So that's why we say during test analysis, the test basis is analyzed to identify testable features and define associated test conditions.



    - During test design, the test conditions are elaborated into high level test cases, sets of high level test cases and other testware.

    During test analysis, we used to ask what to test whereas during test design, we ask how to test. During test design we write test cases based on test condition, along with writing test case, we also identify the defect in the test basis, because while writing test case, we referred test condition and test basis, and during this, if we find any mistake, we shall report it. That is why we say during test design, the test conditions are elaborated into high level test cases, sets of high level test cases and other testware.



    - During test implementation, the testware necessary for test execution is created and completed, including sequencing of the test cases into test procedures.

    As of now, we know during test analysis, we ask what to test, During test design we ask how to test. And now during test implementation, we ask, do we now have everything in place to run the tests?

    First point is testware necessary for test execution is created and completed. To test our requirement, we need computer switch, battery, door and light. 



    So during test implementation, we need to arrange them and check if they are in ready state.

    Second point is sequencing the test cases into test procedures. During test implementation, we sequence the test cases based on priority or risk.



    As you can see here, how test cases are sequenced and this is the order, we will execute them.

    Last point is related to test script, this point is applicable only if you aren't doing automation testing. Here, test cases are converted into test scripts so that they can be automatically executed. That's why we say during test implementation, the testware necessary for test execution is created and completed, including sequencing the test cases into tests procedures.



    - During test execution, test suits are run in accordance with the test execution schedule.

    Here we ask the question, are all tests run? Here we run all the tes suits, test suits consist of set of test cases. Following the test suit example provided in Test Implementation, during test execution, we run them in that order.

    Second point is, after running the test cases, we complete and evaluate the results. Here the expected result is compared with the measured result. That's why we say during test execution test suits are run in accordance with the test execution schedule.



    - Test completion activities collect data from completed test activities to consolidate experience, test-ware and any other relevant information.

    So this is the activity where we collect all the output from the previous activity.






    The activities under test analysis are analyzing the test basis where we analyze the requirement, evaluating the test basis where we have to evaluate the requirement which will tell us what is correct and what is going wrong.

    1. Analyzing the test basis

    First we analyze the requirement specifications and we already covered why we analyze the requirement, is because we are trying to find out what to test. We read the test basis, identify the testable features then finally plan our test conditions.

    The second point is risk analysis reports. Each project will have it’s own risks, the product and project will both have risks and we have to analyze those risks and depending on those risks areas we have to figure out what we need to test. 

    2. Evaluating the test basis.

    First we have to check what has been omitted what is that in the customer requirement but hasn’t been written in our requirement also if there are inconsistencies in the requirement we have to ask the customer for clarification. The same goes for inaccuracies in data sheet or input documents, we must ask the customer to correct them.

    The second point is contradictions in statements. Suppose the customer has given 100 lines of requirements but some lines contradict each other and we need to get clarification from the customer.

    The third point is defining and prioritizing test conditions, what do we need to implement first and what comes next.

    And the final point is bi-directional traceability between test basis and test conditions. We have the requirement and that has to be mapped with our test conditions, this way if the costumer asks us where is the test condition for this particular requirement, we can show him by using this traceability.

    Importance

    Reasons for why test analysis is important

    To reduce the likelihood of omitting important test conditions, to define more precise and accurate test conditions, to produce test conditions which are to be used as test objectives. And another reasons are that test objectives are traceable to the test basis and identification of defects during test analysis.



    1. Designing and prioritising test cases and sets of test cases. 

    In our early example we only have two test cases but in a real life project you will have many such test cases so we have to prioritize which test case we should execute first and this is done in the test design stage, we assign priorities to the test cases.

    2. Identifying necessary test data to support test conditions and test cases.

    Here we have to find necessary data to fulfill these test conditions.

    3. Designing the test environment and identifying any required infrastructure and tools.

    In other words to perform these test cases we need infrastructure 

    4. Capturing bi-directional traceability between the test basis, test conditions and test cases.

    Importance

    Identification of defects in the test basis. This means that when you prepare the test design we refer to previous documents like the test basis and we might identify errors that might not be detected before.

    Find defect in test conditions. Just like before, if you write test cases we study the test conditions and we may come across problems in them.

    Find defect is cost effective.

    Any time we can trace requirement, test condition and test case.



    1. Developing and prioritizing test procedures and potentially creating automates test scripts.

    In the test implementation stage we have to develop the test procedure aswell as the test scripts.



    Suppose that there are these scripts available. Our job is to put them in an order of priority, which test case will come first and which will come after that.

    Definition - Test monitoring involves the ongoing comparison of actual progress against planned progress, using any test monitoring metrics defined in the test plan.

    Are supported by the evaluation of exit criteria, which are referred to as the definition of done in some software development lifecycle models.

    One of the important points why we do test monitoring and control is to communicate information to the stakeholder.

    As we know, test planning draws the road map for the test activity, which includes scheduling, like when testing will start and when it will end. Similarly for all other test activities, but in reality, it is not possible to provide the fixed timing and follow it due to many operational issues. That is the reason it is advised to monitor current status of each test activity against the planned one. So that if there is any lag, we can take necessary action to meet the plant's schedule.

    In test planning stage we decide when the analysis will start and when it will end. While monitoring, we compare the current status of test analysis with the plan schedule.

    Test Control

    Definition - It involves taking necessary actions to meet the objectives of the test plan.

    Evaluation of exit criteria

    Checking -> Test results and logs against specified coverage criteria.

    Assesing -> The level of component or system quality based on test results and logs.

    Determining -> If more tests are needed

    The activities under test analysis are analyzing the test basis where we analyze the requirement, evaluating the test basis where we have to evaluate the requirement which will tell us what is correct and what is going wrong.

    1. Analyzing the test basis

    First we analyze the requirement specifications and we already covered why we analyze the requirement, is because we are trying to find out what to test. We read the test basis, identify the testable features then finally plan our test conditions.

    The second point is risk analysis reports. Each project will have it’s own risks, the product and project will both have risks and we have to analyze those risks and depending on those risks areas we have to figure out what we need to test. 

    2. Evaluating the test basis.

    First we have to check what has been omitted what is that in the customer requirement but hasn’t been written in our requirement also if there are inconsistencies in the requirement we have to ask the customer for clarification. The same goes for inaccuracies in data sheet or input documents, we must ask the customer to correct them.

    The second point is contradictions in statements. Suppose the customer has given 100 lines of requirements but some lines contradict each other and we need to get clarification from the customer.

    The third point is defining and prioritizing test conditions, what do we need to implement first and what comes next.

    And the final point is bi-directional traceability between test basis and test conditions. We have the requirement and that has to be mapped with our test conditions, this way if the costumer asks us where is the test condition for this particular requirement, we can show him by using this traceability.

    The output of the test activity is called work product.

    ISO Standard for Work Product is ISO/IEC/IEEE 29119-3 , this standard only provides the template but the actual implementation depends on the context as we saw in the test process, it varies from organization to organization and product to product.



    The output of test planning is test plans, exit criteria, traceability information



    Test monitoring is done at each test activity and at the end of that test activity we collect the test reports of that stage like test progress reports, test summary reports, and during test monitoring we decide if task is completed, resource is allocated properly and it's used efficiently or if additional effort is required.



    1. Defined and prioritized test conditions

    Along with finding the test condition we also prioritize them.

    2. Creation of test charters

    Charter means documentation of the goal or objective for a test session.

    3. Defects in the test basis

    Discovery and reporting of defects in the test basis.



    1. Test cases and sets of test cases

    As we know during test design we write test cases and therefore test case is the output

    2. Design and/or identification of the test data

    In the test design case we only identify the test data, we don't repair them. Preparation of the test data is done in test implementation stage.

    3. Design of the test environment

    Test environment is designed in the test design stage whereas is implemented in the test implementation stage

    4. Identification of infrastructure and tools



    1. Test procedures and the sequencing of those test procedures

    2. Test suites

    Test suites is collection of test scripts in execution order

    3. A test execution schedule

    4. Service virtualization and automated test scripts

    In some cases test implementation involves creating work products using or used by tools such as service virtualization and automated test scripts

    5. Creation and verification of test data and the test environment

    During test design we only identify test data and test environment

    6. Test conditions defined in test analysis may be further refined in test implementation



    1. Documentation of the status of individual test cases or test procedures (e.g., ready to run, pass, fail, blocked, deliberately skipped, etc.)

    Since the test case is run here we need to document how many test cases are ready to run, how many test cases are passed and failed or how many test cases failed to execute and how many we didn't execute deliberately

    2. Defect reports

    During test execution, test cases are executed and expected result is prepared with measured result and if they are not same then defect is reported

    3. Documentation about which test item(s), test object(s), test tools and testware were involved in the testing

    All the things used test execution are documented and stored so that they can be used again if asked.



    1. Test summary reports

    Here our task is to collect all reports like test summary reports 

    2. Action items for improvement of subsequent projects or iterations

    We need to document any improvement points which we can use for next release

    3. Change requests or product backlog items 

    During test activity if we didn't test any test case or if we didn't execute any test scripts then we need to document them so that we can address them in the next release

    4. Finalized testware

    Everything which was used during testing and all the output are stored

    Traceability is maintained between test basis and test condition and along we have two test cases for a few test conditions so each of these test conditions needs to be traced to test cases (test condition 1 -> test case 1 -> test script 1 -> result). If the customer wants to know how we test a test condition we can show him using traceability.

    Why do we use traceability?

    1. To implement effective test monitoring and control process
    2. Revert to test planning based on the feedback
    3. To communicate to stakeholder

    The importance of traceability?

    1. Analysing the impact of changes

    Suppose there is a change in the requirement then due to traceability we will come to know which work products are affected and we will update them

    2. Making testing auditable

    Since every item is linked is easy while auditing the project

    3. Meeting I.T. governance criteria

    4. Improving the understandability of test progress reports and tests summary reports to include the status of elements of the test basis, for example, requirements that pass their tests, requirements that failed their tests and requirements that have pending tests.

    5. Relating the technical aspects of testing to stakeholders in terms that they can understand

    Since we provide the traceability now, we can generate different types of matrix and provide it to Stakeholders

    6. Providing information to assess product quality, process capability and project progress against business goals

    Since we have different matrixes, we can assess product quality and its process.
