    5.3.1 Metrics Used in Testing

    Test Matrix - Metrics can be collected during and at the end of test activities

    Suppose we are in the test design stage. In this stage, we create the test cases. In the planning stage, it was decided that you need to cover 80% of the requirement. Now, in the test monitoring stage, we will use this matrix for monitoring the progress. Now, suppose test design activity is planned for one month, then two ways we can monitor the progress.

    - first, every week you can check the progress o
    - second way to check once after the months end

    So remember this. Metrics can be collected during and at the end of test activities. 

    Now, let's see some of the common examples of test matrixes:

    1. Percentage of plan work done in test case preparation

    As we saw in this example, where 80% test case preparation was our target. And against this, we were collecting the current matrices for the design phase. 

    2. Percentage of planned work done in test environment preparation

    So during planning we can keep a target like 100% test environment shall be ready before the test execution stage. Then we can monitor its progress through testing activity until the test implementation stage. And if there is the deviation, we can take action. 

    3. Test case execution

    Here we can keep a target, like 75% of the test cases shall execute in the test execution stage. Once the actual test execution happens, we can collect the data and compare against 75%. 

    4. Defect information

    This example is very important, this information is gathered in the form of:

    - defect density
    - defect found and fixed
    - failure rate
    - confirmation test results

    And here, do you think density is defined as the number of defects per unit size of a work product? Let's see this example to understand this. 

    Suppose these are the two work products:

    - test condition from the test analysis stage
    - test cases from the test design stage

    Now, if we take one file from each and find 20 defects in test condition and 25 defects in test cases, then defect density of test case is more.
     
    Definition - Defect density is the number of defects per unit size of a work product.
    Definition - Failure rate is the ratio of the number of failures of a given category to a given unit of measure. 

    Now let's understand this. Suppose the given measure unit is that 20% test cases can fail, this data we get from the planning stage. But while test execution, you got a 40% fail test case. Now, to find the failure rate, you have to find the ratio that is 40 divided by 20. So the failure rate is 2. 

    5. Test coverage of requirements, user stories, acceptance criteria, risks or code

    We already saw an example of this previously. Here, the percentage of test coverage is compared against actual coverage. 

    6. Test completion, resource allocation and usage and effort

    Here we see how much percent of the task is completed against the target, how many resources are involved and their effort so that if we feel a task cannot be completed on time, then we can increase the resource to come back on track. 

    Task Example - Cost of testing, including the test compared to the benefit of finding the next defect or the cost compared to the benefit of running the next test

    5.3.2 Purposes, Contents, and Audiences for Test Reports

    The purpose of test reporting is to summarize and communicate test activity information, both during and at the end of a test activity. 

    Now, you must know the difference between these two terms. Test progress report and test summary report. 

    - When you prepare a test report during the test activity, it is referred to as the Test Progress Report. 
    - And if you prepare a report after the test activity is finished, then it is referred to as a test summary report.

    Test progress report includes:

    - the status of the test activities and progress against the test plan
    - factors impending progress
    - testing planned for the next reporting period
    - the quality of the test object. 

    All these data are collected during the test activity. 

    Test summary report includes:

    - it includes summary of testing performed
    - information on what occurred during a test period
    - deviations from the plan, including deviations in schedule, duration, or effort of test activities
    - status of testing and product quality with respect to the exit criteria or definition of done
    - factors that have blocked or continue to block progress
    - metrics of defects, test cases, test coverage activity progress and resource consumption
    - residual risks
    - reusable test work products produced

    And these data are collected after the test activity is finished.

    5.4 Configuration Management

    Definition - Configuration management is a tool used to establish and maintain the integrity of the component or system, the testware and their relationships to one another through the project and product lifecycle. 



    Suppose you made three release to the customer, the first release was on June 2016, the second release was on October 2018 and the third release was on March 2020. In all these releases you provided software, test report and other supporting documents. But now suppose if the customer comes to you in 2020 and asks for a report which you delivered on June 26 release, then how will you provide it? To address this configuration management tool is used. 

    Now we will understand how configuration management works, to understand this, we will take the same example. In this example, we know the customer is asking for a report from June 2016 release. But the question is, how will you differentiate the report from June 2016? October 2018 and March 2020. To differentiate these working products, we provide a unique I.D. to them, and it is called as versions so that we can identify them uniquely. So when a customer asks for a report, they never ask to show the report for June 2016, they will ask you to give a report for a particular version. Now, all these data will be available in a single tool called Configuration Management Tool. 

    Now, if the customer asks you data from 2016 with the version you can easily provide it using a configuration management tool. This is how the configuration management tool works. 

    Configuration management tools ensures:

    - all test items are uniquely identified, versions controlled, tracked for changes and related to each other
    - all items of test wear are uniquely identified, version controlled, tracked for changes related to each other and related to versions of the test item so the traceability can be maintained throughout the test process
    - All identify documents and software items are referenced unambiguously in test documentation

    5.5 Risks and Testing

    5.5.1 Definition of Risk

    Risk - The possibility of an event in the future, which has negative consequences.

    Likelihood - How frequently this can happen.

    Impact - What harm it will cause.

    What you need to remember is level of risk is determined by likelihood and impact. 

    5.5.2 Product and Project Risks

    Suppose you made a release on 2017, 2018 and 2019 and the final product released in 2020. Now we will see where project release occurs and where product risk. 

    So project risk occurs when you are developing the product. That means before the final delivery, all the risk you have will come under project risk and wants to make the final delivery of the product, then product risk occurs. So if you understand this, it will become very easy for you to remember their examples. So what you need to remember is that any risk that occurs before final delivery is project risk and a risk that occurs after the final delivery is product risk. 

    Product Risk Examples:

    - software might not perform its intended functions according to the specification

    This is the product risk, because if you deliver the product and its software is not working, it has a huge impact.

    - software might not perform its intended functions according to the user, customer or stakeholders needs

    - a system architecture may not adequately support some non-functional requirements

    For example, if you deliver mobile as a product and the response time of it is very slow, then it's a product risk because architecture is not supporting fast response. 

    - a particular computation may be performed incorrectly in some circumstances

    For example,if you deliver a calculator as your product and it's not able to calculate correctly, then it's a product risk. 

    - a loop control structure may be coded incorrectly if this type of bug is there, the delivered product may hang and this is a product risk

    - response times may be inadequate for a high performance transaction processing system

    - user experience feedback might not meet product expectations if we deliver the defective product, then will receive bad feedback and this is a product risk

    Now let's move to project risk. These risks are the risks that occur before delivering the product. 

    Project Risk Examples:

    - Delays may occur in delivery, test completion or satisfaction of exit criteria or definition of don
    - Inaccurate estimates, reallocations of funds to higher priority projects or general cost cutting across the organization may result in inadequate funding
    - Late changes may result in substantial rework

    Organizational Issues Examples:

    - skills training and staff may not be sufficient
    - personnel issues may cause conflict and problems
    - Users, business staff or subject matter experts may not be available due to conflicting business priorities



    Political Issues Examples:

    - testers may not communicate their needs or the test results adequately
    - Developers and or testers may fail to follow up on information found in testing and reviews
    - There may be an improper attitude toward or expectations of testing, for example, not appreciating the value of finding defects during testing

    Technical Issues Examples:

    - requirements may not be defined well enough
    - the requirements may not be met, given existing constraints
    - The test environment may not be ready on time
    - Data conversion, migration planning and their tool support may be late
    - Weaknesses in the development process may impact the consistency or quality of project work product such as design, code, configuration, test data and test cases. 
    - Poor defect management and similar problems may result in accumulated defects and other technical debt

    Supplier Issues Examples:

    - a third party may fail to deliver a necessary product or service or go bankrupt
    - Contractual issues may cause problems to the project.

    5.5.3 Risk-based Testing and Product Quality

    Risk analysis is done:

    - to focus the effort required during testing 
    - to decide where and when to start testing
    - To identify areas that need more attention

    Testing is used:

    - To reduce the probability of an adverse event occurring
    - to reduce the impact of an adverse event
    - to provide information about identified risks
    - to provide information on residual or unresolved risks

    Testing is used as a risk mitigation activity. And when you do testing, based on the risk, it is referred to as a risk-based approach. 

    Why a risk-based approach is used:

    - determine the test techniques to be employed
    - Determined the particular levels and types of testing to be performed, for example security testing, accessibility testing. 
    - Determine the extent of testing to be carried out
    - prioritize testing in an attempt to find the critical defects as early as possible
    - Determine whether any activities in addition to testing could be employed to reduce risk, for example providing training to inexperienced designers

    5.6 Defect Management

    1. When Can a Defect be Detected

    Static testing - A defect can be detected through static testing 

    Example : During the development phase code and design, reviews should be used to detect defects. 

    Dynamic testing - The symptom of the defect, the failure can be detected through dynamic testing

    Example : Test cases are used to detect failures

    During static testing we find defects whereas during dynamic testing we find the failure.

    So which one of these two techniques are more effective with respect to cost aswell to quality? To understand this, first we must understand the following:

    "The earlier a defect is detected and corrected, the lower the cost of quality for the system as a whole"

    So if you detect and correct the defect early in the SDLC then it is more effective regarding to cost aswell to quality. So which method we can apply early, static or dynamic? So if we look into our development lifecycle model we can figure out that until the software is ready we cannot perform dynamic testing, that means for the previous stages you can only perform static testing. So the conclusion is that Static Testing is a cost effective approach to producing high quality software.

    Containment phase

    The defect tracking system should allow the Test Analyst to record the phase in the lifecycle introduced and the phase in which it was found.

    - Requirement Engineering where you will analyse, understand and write your test specifications of the test requirement
    - Review Process, here you will review the requirement document
    - Design Phase and you will review the design document
    - Software Development stage and then you will review the implementation aswell

    So there should be a tool in which you can see where the defect was introduced, for the example if the requirement is wrong then the defect was introduced during the requirement phase. If the requirement is correct and the design is also correct but in Software Development stage while implementing someone did a mistake and that means that the defect was introduced during Software Development stage.

    So what is phase containment? Suppose you are writing the requirement and then you introduce some defect it, while performing the review you found that. That means the defect was introduced and found in the same phase and that is called phase containment.

    Suppose you are in the requirement phase and you introduce the error here but during the first phases you could not find it but during software development while performing the test case execution you found that the system is not behaving like it should behave and after investigating you found the root cause, there was a problem with the requirement engineering and that means we could not achieve the phase containment because the defect was not found in the same phase.

    2. Defect Report



    When writing a defect report you will have to:

    Provide enough information so that the defect report is actionable

    Complete - All the necessary information is in the report
    Concise - No extra information is in the report
    Accurate - The information in the report is correct and clearly states the expected and actual results as well as the proper stepts to reproduce
    Objective - The report is a proffesionally written statement of fact

    Example:

    Non-Functional

    In most of the organizations, most of the time we do Functional testing and for Functional testing the requirements are very clear and it is stated properly but most of the time in Non-functional testing the requirements are not there, but here lets say that you are performing Non-functional testing we have to:

    - provide details regarding the environment (in which environment you are performing the non-functional testing)
    - other performance parameters (e.g. size of the load) (like you are performing web testing then how many users you have)
    - sequence of steps
    - expected results

    Usability Failure

    - It is important to state what the user expected the software to do
    - If the usability standard is that an operation should be completed in less than four mouse clicks, the defect report should state how many clicks were required versus the stated standard

    Standard is not avaliable

    What will you do when standards are not avaliable?

    - The requirements did not cover the non-functional quality aspects of the software, the tester may use the "reasonable person" test to determine that the usability is unacceptable. In that case, the expectations of that "reasonable person" must be clearly stated in the defect report.

    So what are the goals?

    - to obtain a fix for the problem

    So whatever problem you are reporting you want to have a fix for that, so while writing the report you will be more careful and provide all the information which will help the developer to fix the problem.

    - the defect information must also be supplied to support accurate classification

    We should provide enough information like the severity, priority so that the classification can be made easily.

    - risk analysis

    Suppose you find lots of defect in that software, if you report that then it will help the management to understand the risk associated with the software.

    - process improvement

    Whenever you find a defect, it will be analysed and then you will come to know what is the reason we are getting so many defects and after analysis we can improve our process so that we can reduce the defect in next releases or next project.

    3. Defect Classification

    Defect Status

    Defect Status or Bug Status in defect life cycle is the present state from which the defect or a bug is currently undergoing. The goal of defect status is to precisely convey the current state or progress of a defect or bug in order to better track and understand the actual progress of the defect life cycle.
    The number of states that a defect goes through varies from project to project.


    New: When a new defect is logged and posted for the first time. It is assigned a status as NEW.
    Assigned: Once the bug is posted by the tester, the lead of the tester approves the bug and assigns the bug to the developer team
    Open: The developer starts analyzing and works on the defect fix
    Fixed: When a developer makes a necessary code change and verifies the change, he or she can make bug status as "Fixed."
    Pending retest: Once the defect is fixed the developer gives a particular code for retesting the code to the tester. Since the software testing remains pending from the testers end, the status assigned is "pending retest."
    Retest: Tester does the retesting of the code at this stage to check whether the defect is fixed by the developer or not and changes the status to "Re-test."
    Verified: The tester re-tests the bug after it got fixed by the developer. If there is no bug detected in the software, then the bug is fixed and the status assigned is "verified."
    Reopen: If the bug persists even after the developer has fixed the bug, the tester changes the status to "reopened". Once again the bug goes through the life cycle.
    Closed: If the bug is no longer exists then tester assigns the status "Closed." 
    Duplicate: If the defect is repeated twice or the defect corresponds to the same concept of the bug, the status is changed to "duplicate."
    Rejected: If the developer feels the defect is not a genuine defect then it changes the defect to "rejected."
    Deferred: If the present bug is not of a prime priority and if it is expected to get fixed in the next release, then status "Deferred" is assigned to such bugs
    Not a bug:If it does not affect the functionality of the application then the status assigned to a bug is "Not a bug".


    Classification information for newly identified defects

    1. Project Activity - That resulted in the defect being detected - e.g. review, audit, inspection, coding, testing
    2. Project Phase - Defect was introduced (if known) - e.g. requirements, design, detailed design, code
    3. Project Phase - Defect was detected - e.g. requirements, design, detailed design, code, code review, unit test, integration test, system test, acceptance test
    4. Suspected cause of defect - Requirements, design, interface, code, data
    5. Repeatability - Once, intermittent, reproducible
    6. Symptom - Symptom - e.g. crash, hang, user interface error, system error, performance

    Classification after investigation of defect

    So after you investigate the error you can do further classifications.

    1. Root Cause - The mistake that was made that resulted in the defect (process, coding error, user error, test error, configuration issue, data issue, third party software, external software problem)
    2. Source - The work product in which the mistake was made (requirements, design, detailed design, architecture, database design, user documentation, test documentation)
    3. Type - Logic problem, computational problem, timing issue, data handling, enhancement
    4. Resolution (once the defects are fixed) - Code change, documentation change, deffered, not a problem, duplicate
    Resolution means, you had to do a code change to change the resolution or you only did documentation change and everything is fine now.
    5. Corrective Action - Requirements review, code review, unit test, configuration documentation, data preparation, no change made

    Additional Classification

    Classification depends on the organization because each project is different and each organization is working in a different way.

    1. Severity and Priority
    2. Safety Impact
    3. Project schedule impact
    4. Project costs
    5. Project risk
    6. Project quality impact 

    Final Resolution

    The Final Resolution can be fixed/verified, closed/not a problem, open/unresolved
